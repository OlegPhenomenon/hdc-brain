Экспедиция IV Глава 15

HDCBlock — собираем блок целиком

Вспомним откуда мы пришли.

В последних главах мы разобрали три отдельных механизма:

  • HDCMemory (глава 12) — "что было раньше?" — взвешенное накопление прошлого
  • MultiHeadBindingAttention (глава 14) — "на кого обратить внимание?" — 12K параметров
  • ControllerBlock (глава 3) — "подумать над результатом" — 21M параметров

Теперь нужно собрать их вместе в один рабочий блок.

Строительство дома: аналогия

Представь что строишь дом.

Сначала заливаешь фундамент (Memory — стабильная база из прошлого). Потом строишь стены (Attention — структура). Потом делаешь отделку внутри (Controller — думаешь, обрабатываешь).

Но важная деталь: каждый этап оставляет то что было. Фундамент никуда не девается. Стены строятся поверх фундамента, но фундамент остаётся. Отделка — поверх стен.

Это называется residual connection (мы разбирали в главе 3).

Поток информации через HDCBlock

  

    CODE
    
  

  Вход x: (B, T, 4096)
         │
         ▼
┌─────────────────┐
│   HDCMemory     │  "что было в прошлом?"
│   mass + decay  │
└────────┬────────┘
         │ mem_output
         ▼
   x = LayerNorm(x + mem_output)   ← residual + нормализация
         │
         ▼
┌────────────────────────┐
│ MultiHeadBindingAttn   │  "на кого обратить внимание?"
│ 4 heads, sigmoid       │
└────────────┬───────────┘
             │ attn_output
             ▼
   x = LayerNorm(x + attn_output)  ← residual + нормализация
             │
             ▼
┌─────────────────┐
│  ControllerBlock│  "подумать"
│  4096→2560→4096 │
└────────┬────────┘
         │
         ▼
  Выход: (B, T, 4096)  ← та же форма что вход

Два паттерна повторяются постоянно:

  1. Residual: x = x + output_подблока — сохраняем всё что было
  2. LayerNorm: нормализуем перед следующим шагом — стабилизируем числа

Pre-LN: нормализация ДО подблока

В оригинальном Transformer 2017 года нормализация шла после подблока:

  

    CODE
    
  

  Post-LN:  x → Sublayer → x + result → LayerNorm → output

Это работало, но нестабильно для очень глубоких сетей.

В GPT-2 (2019) переключились на Pre-LN — нормализация до:

  

    CODE
    
  

  Pre-LN:   x → LayerNorm → Sublayer → x + result → output

Почему лучше?

При обучении нейросети важно чтобы числа не "взрывались" (не становились очень большими) и не "схлопывались" (не становились очень маленькими). LayerNorm перед каждым блоком гарантирует: вход всегда нормализован — стабильно обучается даже на 100+ слоях.

Наш код использует Pre-LN:

  

    PYTHON
    
  

  def forward(self, x):
    mem = self.memory(x)
    x = self.ln_mem(x + mem)      # residual, потом норм ← Pre-LN!
    attn = self.attention(x)
    x = self.ln_attn(x + attn)    # residual, потом норм ← Pre-LN!
    x = self.controller(x)        # controller сам содержит LN внутри
    return x

Полный код HDCBlock

  

    PYTHON
    
  

  import torch
import torch.nn as nn

class HDCBlock(nn.Module):
    def __init__(self, hdc_dim, controller_dim, n_heads=4, dropout=0.1):
        super().__init__()

        # Три основных компонента
        self.memory     = HDCMemory(hdc_dim)
        self.attention  = MultiHeadBindingAttention(hdc_dim, n_heads)
        self.controller = ControllerBlock(hdc_dim, controller_dim, dropout)

        # Нормализация после memory и attention
        self.ln_mem  = nn.LayerNorm(hdc_dim)
        self.ln_attn = nn.LayerNorm(hdc_dim)
        # ControllerBlock сам содержит LayerNorm внутри — поэтому здесь нет третьего

    def forward(self, x):
        # === Memory: узнаём что было раньше ===
        mem = self.memory(x)
        x = self.ln_mem(x + mem)    # добавляем к текущему x, нормализуем

        # === Attention: выбираем на что обратить внимание ===
        attn = self.attention(x)
        x = self.ln_attn(x + attn)  # снова residual + нормализация

        # === Controller: думаем ===
        x = self.controller(x)      # 4096 → 2560 → GELU → 4096 → residual

        return x

Параметры одного блока

Посчитаем честно:

  

    CODE
    
  

  HDCMemory:
  mass_proj  = 4096 × 1 = 4,096
  decay_proj = 4096 × 1 = 4,096
  Итого:                  8,192   (0.04% блока)

MultiHeadBindingAttention:
  bv_q + bv_k + bv_v = 3 × 4 × 1024 = 12,288   (0.06% блока)

ControllerBlock:
  ln:              4096 × 2         =  8,192
  fc_down:         4096 × 2560      = 10,485,760
  fc_up:           2560 × 4096      = 10,485,760
  ln2 (residual):  4096 × 2         =  8,192
  Итого:                            = 20,987,904  (99.9% блока!)

LayerNorm (×2, внутри HDCBlock):
  2 × 4096 × 2 = 16,384

─────────────────────────────────────────────────
Итого 1 HDCBlock: ≈ 21,024,768 ≈ 21M параметров

Интересный факт: ControllerBlock занимает 99.9% параметров блока.

Memory — 0.04%. Attention — 0.06%. Controller — 99.9%.

Это не ошибка! Это дизайнерское решение. В нашей архитектуре:

  • Memory и Attention работают дёшево — собирают контекст
  • Controller работает дорого — глубоко обрабатывает информацию

Восемь блоков: почему именно 8?

В модели 8 блоков.

  

    CODE
    
  

  8 блоков × 21M параметров = 168M параметров

Это 56% всей модели (из 299M).

Почему 8, а не 4 или 16?

  

    CODE
    
  

  4 блока:  4 × 21M = 84M    ← маловато глубины
8 блоков: 8 × 21M = 168M   ← наш выбор (+ 3 мысли = 24 эффективных прохода)
16 блоков: 16 × 21M = 336M ← больше чем вся модель должна быть

Плюс: у нас 3 мысли (ThoughtLoop, следующая глава). Каждая мысль — полный проход через все 8 блоков. Итого: 8 × 3 = 24 "эффективных" прохода через данные.

У SmolLM2: 32 слоя, 1 проход. У нас: 8 блоков, 3 прохода = 24 "эффективных слоя". Не так уж плохо.

Gradient Checkpointing: техническая деталь

В коде есть важная настройка:

  

    PYTHON
    
  

  # В конфиге модели:
USE_CHECKPOINT = True

Что это делает?

При обычном forward pass PyTorch сохраняет в памяти все промежуточные вычисления:

  

    CODE
    
  

  x → Block1 → a1 → Block2 → a2 → Block3 → a3 → ... → loss
         [сохр]        [сохр]        [сохр]  ...

Зачем сохранять? Для backpropagation нужно знать "где были" при прямом проходе.

При 8 блоках, батче 12, длине 512, размерности 4096 и bf16 — это ~500 МБ только активации.

С gradient checkpointing:

  

    CODE
    
  

  x → Block1 → (НЕ сохраняем a1) → Block2 → ...

Когда при backprop нужна a1 — пересчитываем Block1 заново.

Цена: в 2 раза больше вычислений. Выгода: в 8 раз меньше памяти на активации.

Для нас это необходимо: без checkpoint batch=12 не влезает в 24GB GPU.

  

    PYTHON
    
  

  from torch.utils.checkpoint import checkpoint as torch_checkpoint

def forward_blocks(h, blocks):
    for block in blocks:
        if USE_CHECKPOINT and training:
            h = torch_checkpoint(block, h, use_reentrant=False)
        else:
            h = block(h)
    return h

Что происходит в разных блоках

Теоретически каждый блок должен "специализироваться" на чём-то своём. Исследования трансформеров показывают такую картину:

  

    CODE
    
  

  Блок 1-2 (ранние):
  Синтаксис — части речи, базовые зависимости
  "это глагол", "это существительное"

Блок 3-5 (средние):
  Семантика — значения слов, тематические связи
  "это слово о Франции", "это слово о движении"

Блок 6-8 (поздние):
  Прагматика — контекст, предсказание
  "что вероятнее всего идёт следующим?"

Но в нашей архитектуре с ThoughtLoop всё сложнее — каждая из 3 "мыслей" проходит все 8 блоков. Первая мысль — базовый анализ, вторая — уточнение, третья — финальное предсказание. Но это тема следующей главы.

✅ Итоги главы

HDCBlock — комбинация трёх механизмов:

  

    CODE
    
  

  x → [HDCMemory] → residual + LN → [BindingAttention] → residual + LN → [Controller] → output

Memory:     8,192 параметров (0.04%)   — "что было"
Attention: 12,288 параметров (0.06%)   — "на что смотреть"
Controller: 21M параметров  (99.9%)    — "думать"

Ключевые паттерны:

  • Residual: x = x + sublayer_output — сохраняем информацию
  • Pre-LN: нормализация перед каждым подблоком — стабильное обучение
  • 8 блоков: 168M параметров (56% модели), умножается на 3 мысли
