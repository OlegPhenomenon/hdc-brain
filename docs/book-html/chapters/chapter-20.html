Экспедиция V Глава 20

Эффективное обучение на GPU

Представь что ты строишь дом.

Есть два варианта: наймёшь одного мастера на все руки — или бригаду из ста рабочих.

Один мастер — умный, опытный, делает всё последовательно: сначала фундамент, потом стены, потом крыша. Одна задача в момент времени.

Бригада из ста рабочих — каждый делает одно простое действие. Один несёт кирпич, другой накладывает раствор, третий ровняет. Одновременно. Никто не ждёт.

Обучение нейронной сети — это про миллиарды простых операций. CPU — умный мастер. GPU — бригада из десяти тысяч рабочих.

Для нашей задачи нужна бригада.

Почему нейросети любят GPU

Посмотрим на один слой нашей модели — ControllerBlock:

  

    CODE
    
  

  Матрица W1: размер (4096, 2560) = 10,485,760 чисел
Умножение батча: (12, 512, 4096) × (4096, 2560) = 64 миллиарда операций

64 миллиарда операций за один проход через один слой.

  

    CODE
    
  

  CPU:
  Ядер: 16 (мощный сервер)
  Операций/сек: ~100 миллиардов
  Время: 64B / 100B = 0.64 секунды за слой

GPU (RTX 3090):
  "Ядер" (CUDA cores): 10,496
  Операций/сек: 142,000 миллиардов (TFLOPS) для bf16
  Время: 64B / 142,000B = 0.00045 секунды за слой

Разница: в 1,400 раз быстрее.

GPU делает одно: берёт большие таблицы чисел и перемножает их миллиардами параллельных потоков. Именно это нужно нейросети.

Главная проблема: не скорость, а память

Вот неожиданная вещь: GPU у нас достаточно быстрый.

Главное ограничение — VRAM (видеопамять). Наш сервер — RTX 3090 с 24 ГБ.

Посчитаем что занимает память:

  

    CODE
    
  

  Параметры модели (bf16):
  299M параметров × 2 байта = 598 МБ ≈ 0.6 ГБ

Состояния оптимизатора AdamW (float32):
  299M × 2 (m и v) × 4 байта = 2,392 МБ ≈ 2.4 ГБ

Активации при forward pass (batch=16, seq=512, bf16):
  Каждый из 8 блоков сохраняет промежуточные результаты:
  16 × 512 × 4096 × 8 блоков × 2 байта ≈ 3.4 ГБ
  × 3 мысли = 10.2 ГБ!

Градиенты:
  ≈ 0.6 ГБ

─────────────────────────────
Итого БЕЗ оптимизаций: ~14 ГБ
Итого С checkpointing:  ~7 ГБ

RTX 3090 имеет 24 ГБ. Без оптимизаций — тесновато. С оптимизациями — работает.

Главный "пожиратель" памяти — активации при 3 мыслях: 10 ГБ только на них!

Gradient Checkpointing: платим временем, экономим память

Вернёмся к тому что мы уже знаем из главы 15.

При обычном forward pass PyTorch запоминает всё:

  

    CODE
    
  

  x → Block1 → [запомнили a1] → Block2 → [запомнили a2] → ... → loss

Зачем запоминать? При backpropagation нужно знать "где были" чтобы посчитать градиенты.

Gradient Checkpointing говорит: "Не запоминай! Когда понадобится — пересчитаем заново."

  

    PYTHON
    
  

  from torch.utils.checkpoint import checkpoint as torch_checkpoint

# Вместо:
a1 = block(x)  # ← результат запоминается в памяти

# Используем:
a1 = torch_checkpoint(block, x, use_reentrant=False)
# ← результат НЕ запоминается, при backprop пересчитается

Цена: каждый блок вычисляется дважды (forward + пересчёт при backprop). Выгода: в 8 раз меньше памяти на активации.

  

    CODE
    
  

  Без checkpointing: 10.2 ГБ активаций
С checkpointing:   1.3 ГБ активаций (в 8× меньше)
Вычислений больше: в 2× (каждый блок пересчитывается)

Для нашей задачи — стоит того. Нам важнее поместиться в памяти, чем сэкономить 2× на вычислениях.

История с torch.compile: 73 минуты зависания

PyTorch 2.0 добавил torch.compile — волшебную кнопку ускорения:

  

    PYTHON
    
  

  model = torch.compile(model)  # компилирует в быстрый CUDA-код
# Обычный результат: в 1.5—2.5× быстрее!

Мы попробовали. Процесс запустился, GPU загрузился на 100%...

И не вывел ни единой строки лога 73 минуты.

Что случилось?

torch.compile создаёт статичный граф вычислений: компилирует один раз, потом переиспользует. Это работает только если граф не меняется.

Наш ThoughtLoop содержит Python-цикл:

  

    PYTHON
    
  

  for t in range(1, n_thoughts):  # ← Python цикл!
    # ...

При изменении n_thoughts граф меняется. CUDA перекомпилирует. Потом опять. Потом опять. Бесконечный цикл перекомпиляций.

  

    PYTHON
    
  

  # В итоге в train.py:
# torch.compile disabled: causes infinite recompilation with 3 thoughts
# ThoughtLoop has dynamic control flow that breaks CUDA graphs
print("torch.compile: OFF (incompatible with Thought Loops)")

Если захочется включить compile в будущем — нужно развернуть цикл вручную:

  

    PYTHON
    
  

  # Вместо for-цикла — три явные функции без ветвления
def forward_thought_1(h, blocks): ...
def forward_thought_2(h, blocks, gate2, pos2): ...
def forward_thought_3(h, blocks, gate3, pos3): ...

Тогда граф будет статичным. Но это пока не сделано — оставлено как будущая оптимизация.

Урок: "работает везде" и "работает оптимально" — разные вещи. Всегда проверяй что оптимизация реально работает, смотри логи.

HDC Binding Attention: случайная экономия памяти

В стандартном трансформере attention создаёт матрицу (T × T):

  

    CODE
    
  

  scores = Q @ K.T  # форма: (B, H, T, T)
# Для T=512, H=4, B=16:
# 16 × 4 × 512 × 512 = 16,777,216 элементов = 128 МБ на батч

Есть специальная оптимизация — FlashAttention — которая вычисляет это по кускам и никогда не хранит полную матрицу.

Наш HDC Binding Attention случайно избегает этой проблемы:

  

    PYTHON
    
  

  # Вместо Q @ K.T (матрица T×T):
binding = q * k  # поэлементное умножение! форма: (B, H, T, D//H)

Никакой матрицы T×T. FlashAttention нам просто не нужен.

Это не намеренная оптимизация — это следствие того как работает HDC Binding (глава 14). Но побочный эффект приятный: меньше памяти.

Батч и накопление: как выбрать размер

Большой батч — хорошо. Модель видит много примеров за раз, градиенты стабильные.

Но память не резиновая.

  

    CODE
    
  

  batch=128, seq=512: требует ~18 ГБ активаций → не влезает в 24 ГБ
batch=16,  seq=512: требует ~2.3 ГБ активаций → влезает с запасом

Решение — gradient accumulation (глава 19): делаем 8 шагов с batch=16, накапливаем градиенты, потом один шаг оптимизатора.

  

    CODE
    
  

  8 × batch=16 = эффективный batch=128

Эффект для скорости:

  

    CODE
    
  

  Один шаг = 8 форвардов + 8 backward + 1 optimizer.step()
Время на шаг: ~500ms на RTX 3090
Токенов за шаг: 16 × 512 × 8 = 65,536
Скорость: 65,536 / 0.5 ≈ 131,000 токенов/сек

За 40,000 шагов × 65K токенов = 2.6 миллиарда токенов обучения.

GPU не должен ждать данных

Есть незаметная ловушка: GPU иногда ждёт CPU.

  

    CODE
    
  

  Плохой сценарий:
GPU: [считает........] [ждёт данных] [считает........] [ждёт] ...
CPU:              [загружает данные] [загружает данные]
GPU простаивает ~40% времени!

Причина: CPU медленно читает файл с диска, пока GPU уже закончил предыдущий батч.

  

    PYTHON
    
  

  from torch.utils.data import DataLoader

loader = DataLoader(
    dataset,
    batch_size=BATCH_SIZE,
    num_workers=4,     # 4 отдельных процесса CPU загружают данные параллельно
    pin_memory=True,   # данные в "закреплённой" памяти → быстрее перенос на GPU
    prefetch_factor=2, # загружают следующие 2 батча заранее
)

num_workers=4 — пока GPU считает батч N, 4 CPU-процесса уже загружают батч N+1 и N+2.

pin_memory=True — данные помещаются в специальную память которую нельзя выгрузить на диск. GPU читает её напрямую, без лишнего копирования.

Результат:

  

    CODE
    
  

  Хороший сценарий:
GPU: [считает........][считает........][считает........]
CPU: [загрузка][загрузка][загрузка][загрузка]
GPU занят 95% времени!

bfloat16: скорость без жертв

Всё обучение идёт в bfloat16 (bf16). Уже говорили в главе 19, но здесь важны практические детали.

  

    PYTHON
    
  

  # Инициализация:
model = model.to(device)  # параметры переезжают на GPU
# По умолчанию float32 — 4 байта на число

# Включаем bf16 во время вычислений:
with torch.autocast(device_type='cuda', dtype=torch.bfloat16):
    logits, loss = model(x, y)
# Матрицы умножаются в bf16 → в 2.5× быстрее
# loss.backward() → PyTorch сам разбирается где нужна точность float32

Что "autocast" делает автоматически:

  

    CODE
    
  

  Матричные умножения (nn.Linear): bf16 ← самые частые операции
LayerNorm, softmax: float32 ← нужна точность
Накопление градиентов: float32 ← нельзя терять точность
Adam state (m, v): float32 ← критически важная точность

Для float16 (не bfloat16) нужен дополнительный GradScaler — умножает gradients на большое число чтобы они не исчезли (underflow). Для bf16 это не нужно: диапазон чисел как у float32.

Мониторинг: что смотреть пока идёт обучение

Обучение на 24 часа — это не "запустил и забыл". Нужен мониторинг.

  

    BASH
    
  

  # В отдельном терминале:
watch -n 2 nvidia-smi

  

    CODE
    
  

  +------------------------------------------------------------------+
| GPU  Name          | Temp | Perf | Pwr: Usage  | Memory-Usage    |
+------------------------------------------------------------------+
|  0  RTX 3090       |  74C | P2   |  335W/350W  | 20847MiB/24576MiB |
+------------------------------------------------------------------+
| GPU-Util: 96%  |

Хорошие признаки:

  • GPU-Util: 90-100% — GPU загружен, не ждёт
  • Memory: 80-90% — использует хорошо, есть небольшой запас
  • Power: близко к максимуму — GPU работает на полной мощности
  • Temperature: до 83°C — норма (предел для RTX 3090 — 93°C)

Плохие признаки:

  • GPU-Util: 40-60% — CPU не успевает подавать данные
  • Memory: 100% → скоро OutOfMemory
  • Logfile не обновляется 10+ минут → процесс завис (см. историю с torch.compile)

Сохранение прогресса: чекпойнты

Обучение 24 часа может прерваться по любой причине: упал сервер, закончилась аренда, баг в коде.

Если не сохранять — потеряешь всё.

  

    PYTHON
    
  

  # Каждые SAVE_INTERVAL шагов:
if step % SAVE_INTERVAL == 0 and step > 0:
    checkpoint = {
        'step': step,
        'model_state': model.state_dict(),       # веса модели
        'optimizer_state': optimizer.state_dict(), # состояние Adam (m и v!)
        'loss': current_loss,
        'bpb': current_bpb,
    }
    path = f'checkpoint_{step:06d}.pt'
    torch.save(checkpoint, path)
    print(f"Сохранён чекпойнт: {path}")

Критически важно: сохраняем и model_state, и optimizer_state.

Почему optimizer_state? Adam хранит для каждого параметра m и v (история градиентов). Если перезапустить с нуля — Adam "забудет" всю историю и первые 1000 шагов будет нестабильным.

Загрузка и продолжение:

  

    PYTHON
    
  

  ckpt = torch.load('checkpoint_010000.pt', map_location=device)
model.load_state_dict(ckpt['model_state'])
optimizer.load_state_dict(ckpt['optimizer_state'])
start_step = ckpt['step']
print(f"Продолжаем с шага {start_step}, loss={ckpt['loss']:.3f}")

Обычно сохраняем каждые 500 шагов и храним последние 3 чекпойнта (старые удаляем).

Задание 20.1 — Память

Посчитай сколько памяти занимает модель в разных конфигурациях:

  

    PYTHON
    
  

  import torch

def memory_estimate(n_params, batch, seq, n_blocks, n_thoughts, dtype_bytes=2):
    """
    n_params: количество параметров
    batch, seq: размер батча и последовательности
    n_blocks: количество блоков
    n_thoughts: количество мыслей
    dtype_bytes: 2 для bf16/fp16, 4 для fp32
    """
    params_mb = n_params * dtype_bytes / 1024**2
    # Состояния оптимизатора Adam (float32, 2 тензора: m и v):
    adam_mb = n_params * 2 * 4 / 1024**2
    # Активации (упрощённо: batch × seq × dim × n_blocks):
    dim = 4096
    activations_mb = batch * seq * dim * n_blocks * n_thoughts * dtype_bytes / 1024**2

    total = params_mb + adam_mb + activations_mb
    print(f"Параметры: {params_mb:.0f} МБ")
    print(f"Оптимизатор: {adam_mb:.0f} МБ")
    print(f"Активации: {activations_mb:.0f} МБ")
    print(f"Итого: {total:.0f} МБ = {total/1024:.1f} ГБ")
