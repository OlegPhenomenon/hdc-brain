Экспедиция IV Глава 12

HDC Memory — как модель помнит прошлое

Ира — архивариус в библиотеке.

К ней приходят люди и просят помощи: "Ищу книгу про французскую революцию".

Ира умная. Она не просто смотрит последнюю книгу что принесли — она помнит весь архив. Но помнит по-разному.

Книгу которую принесли вчера — помнит отлично. Книгу из 1990 года — помнит хуже. А книга про котов из 1975 года — вообще неважна для запроса про революцию.

У Иры два фильтра:

  1. Насколько давно (время влияет на память — новее = лучше помним)
  2. Насколько важно (содержание: книга про Францию важнее книги про котов)

Проблема без памяти

После кодбука (глава 11) мы имеем матрицу векторов:

  

    CODE
    
  

  Строка 0: [вектор "The"]
Строка 1: [вектор "capital"]
Строка 2: [вектор "of"]
Строка 3: [вектор "France"]
Строка 4: [вектор "is"]

Каждый вектор — в полной изоляции. Токен "is" на позиции 4 понятия не имеет что было на позициях 0-3.

Но чтобы предсказать что идёт после "is" — нужно помнить "France" и "capital". Иначе это "the [что-то] of France is" может быть чем угодно.

Нужна память — механизм который даёт каждому токену доступ к прошлым.

Простейшая идея: просто сложить все прошлые токены

  

    PYTHON
    
  

  # Для токена на позиции t:
# context[t] = token[0] + token[1] + ... + token[t]

context = torch.cumsum(tokens, dim=1)  # накапливаем суммы

Это работает? Немного. Проблема: "the" и "France" получают одинаковый вес. Но "France" несёт намного больше информации для предсказания "Paris".

Нужна взвешенная память. Важные токены — больший вклад. Старые токены — меньший.

Mass: насколько ты важен?

Первый компонент HDCMemory — масса (mass).

Для каждого токена вычисляем число от 0 до 1: насколько этот токен "весомый"?

  

    PYTHON
    
  

  # Один обучаемый линейный слой: 4096 чисел → 1 число
self.mass_proj = nn.Linear(hdc_dim, 1, bias=False)

# В forward:
mass = torch.sigmoid(self.mass_proj(tokens))  # (B, T, 1)
# sigmoid превращает любое число в диапазон [0, 1]

sigmoid — это S-образная функция:

  

    CODE
    
  

  sigmoid(-5) ≈ 0.01  (почти 0)
sigmoid( 0) = 0.50  (ровно половина)
sigmoid(+5) ≈ 0.99  (почти 1)

Модель сама учится присваивать массу:

  • "the" → mass ≈ 0.03 (артикль, мало информации)
  • "France" → mass ≈ 0.92 (существительное, несёт смысл)
  • "capital" → mass ≈ 0.85 (тоже важное)

Это обучается автоматически через backpropagation.

Decay: ты помнишь хуже то что давно

Второй компонент — затухание (decay).

Даже важные токены со временем "тускнеют". Слово из 5 позиций назад менее важно чем слово из вчера.

  

    PYTHON
    
  

  # Ещё один обучаемый линейный слой
self.decay_proj = nn.Linear(hdc_dim, 1, bias=False)

# В forward:
decay = torch.sigmoid(self.decay_proj(tokens))  # (B, T, 1)
# decay ≈ 0.0 → этот токен быстро забудется
# decay ≈ 1.0 → этот токен запомнится надолго

Снова: модель сама решает. Токен "France" может получить высокий decay (долго помним) и высокую mass (сильно весомый). Токен "the" — низкий decay (быстро забываем) и низкую mass.

Как decay работает через время

Представим три токена: "The" (pos=0), "capital" (pos=1), "France" (pos=2).

  

    CODE
    
  

  decay[0] = 0.9   (для "The")
decay[1] = 0.8   (для "capital")
decay[2] = 0.95  (для "France")

Когда мы на позиции 4 смотрим на прошлые токены:

  

    CODE
    
  

  Сила токена 0 ("The"):     decay[0] × decay[1] × decay[2] × decay[3] = 0.9 × 0.8 × 0.95 × ...
Сила токена 1 ("capital"): decay[1] × decay[2] × decay[3] = 0.8 × 0.95 × ...
Сила токена 2 ("France"):  decay[2] × decay[3] = 0.95 × ...

Чем дальше токен — тем больше decay-множителей перемножено — тем слабее сигнал.

Накопленная память через cumsum

Теперь как это эффективно посчитать?

Для каждой позиции нужна сумма всех предыдущих токенов с правильными весами:

  

    CODE
    
  

  context[t] = Σ (накопленный_decay[i→t] × mass[i] × token[i])
             i≤t

Наивно — это вложенный цикл O(T²). Для T=512 — медленно.

Решение: логарифмическое пространство + cumsum (накопительная сумма):

  

    PYTHON
    
  

  # log(a × b × c) = log(a) + log(b) + log(c)
# Произведение decay → сумма log(decay)

log_decay = torch.log(decay.clamp(min=1e-6))   # (B, T, 1)
cum_log = torch.cumsum(log_decay, dim=1)        # (B, T, 1)
# cum_log[t] = log(decay[0]) + log(decay[1]) + ... + log(decay[t])
#            = log(decay[0] × decay[1] × ... × decay[t])

Разница между двумя позициями:

  

    PYTHON
    
  

  # Затухание от позиции i до позиции t:
# = decay[i+1] × decay[i+2] × ... × decay[t]
# = exp(cum_log[t] - cum_log[i])

diff = cum_log - cum_log.transpose(1, 2)  # (B, T, T)
# diff[t, i] = cum_log[t] - cum_log[i] = log(затухание i→t)

Полный код HDCMemory

  

    PYTHON
    
  

  def forward(self, tokens_bound):
    B, T, D = tokens_bound.shape

    # === Шаг 1: Считаем mass и decay ===
    mass  = torch.sigmoid(self.mass_proj(tokens_bound))    # (B, T, 1)
    decay = torch.sigmoid(self.decay_proj(tokens_bound))   # (B, T, 1)

    # Взвешенные токены (масса определяет "важность")
    weighted = mass * tokens_bound                          # (B, T, D)

    # === Шаг 2: Переходим в log-пространство ===
    log_decay = torch.log(decay.clamp(min=1e-6))           # (B, T, 1)
    cum_log = torch.cumsum(log_decay, dim=1)               # (B, T, 1)

    # === Шаг 3: Матрица затуханий между позициями ===
    diff = cum_log - cum_log.transpose(1, 2)               # (B, T, T)
    diff = diff.clamp(max=0)                               # только затухание, не усиление

    # === Шаг 4: Causal mask — видим только прошлое ===
    causal = torch.tril(torch.ones(T, T, device=tokens_bound.device))
    diff = diff.masked_fill(causal == 0, -1e9)             # будущее → -inf

    # === Шаг 5: Матрица весов ===
    W = torch.exp(diff)                                    # (B, T, T)
    # W[b, t, i] = насколько токен i виден из позиции t
    # Если i > t → W = exp(-inf) ≈ 0 (будущее не видно)
    # Если i = t → W = exp(0) = 1
    # Если i < t → W = exp(отрицательное) < 1 (затухание)

    # === Шаг 6: Взвешенная сумма ===
    context = torch.matmul(W, weighted)                    # (B, T, D)

    return context

Визуализация матрицы W

Матрица W (T×T) показывает "кто сколько помнит кого":

  

    CODE
    
  

           token 0  token 1  token 2  token 3  token 4
  pos 0 [ 1.00,   0.00,   0.00,   0.00,   0.00 ]  ← видит только себя
  pos 1 [ 0.85,   1.00,   0.00,   0.00,   0.00 ]  ← себя + token 0 с затуханием
  pos 2 [ 0.72,   0.85,   1.00,   0.00,   0.00 ]
  pos 3 [ 0.61,   0.72,   0.85,   1.00,   0.00 ]
  pos 4 [ 0.52,   0.61,   0.72,   0.85,   1.00 ]

Ключевые свойства:

  • Нижнетреугольная — каждый токен видит только себя и прошлое (causal)
  • Диагональ = 1.0 — сам себя помнишь идеально
  • Убывает влево — дальние токены помнятся хуже

Но если у токена высокая mass, его вклад больше — даже если он далеко:

  

    CODE
    
  

           token 0  token 1  "France"  token 3
  pos 3 [ 0.03,   0.04,    0.85,    1.00 ]
         ↑ "the"            ↑ mass=0.95 × decay
         низкая mass          → сильный вклад!

Сколько параметров занимает вся HDCMemory?

  

    CODE
    
  

  mass_proj:  4096 × 1 = 4,096 параметров
decay_proj: 4096 × 1 = 4,096 параметров

ИТОГО: 8,192 параметра

8 тысяч параметров. Это 0.003% одного блока.

Для сравнения: ControllerBlock в том же блоке → 21 миллион параметров.

HDCMemory — невероятно компактный механизм который даёт модели доступ ко всему прошлому контексту.

Почему так мало? Потому что мы не создаём "пространство памяти" — мы просто взвешиваем сами токены. Хитрость в том, что весовые коэффициенты (mass и decay) зависят от содержания токена, а не от его позиции.

Связь с Mamba

Если вы читали про архитектуры без трансформеров — возможно слышали про Mamba (2023).

Mamba — это "selective state space model". Она тоже запоминает прошлое и сама решает что важно запомнить.

Сравнение:

  

    CODE
    
  

  Mamba:       h_t = A_t × h_{t-1} + B_t × x_t
             y_t = C_t × h_t
             (A, B, C — зависят от данных)

HDCMemory:   context_t = Σ exp(cumlog[t] - cumlog[i]) × mass[i] × token[i]
             (decay и mass — тоже зависят от данных)

Математически разные. Концептуально — одна идея:

  • Накопление прошлого
  • Data-dependent gates (наши mass/decay = гейты Mamba)
  • Параллельный вычисление через cumsum

Разница: в Mamba есть отдельный "скрытый вектор" h — внутреннее состояние. У нас нет — мы просто взвешиваем сами токены. Проще, меньше параметров.

Задания

Задание 12.1 — Реализуй наивно

Реализуй простую HDCMemory без log-space трюка, через обычный цикл:

  

    PYTHON
    
  

  def naive_memory(tokens, mass, decay):
    """
    tokens: (T, D)
    mass:   (T, 1)  — важность токена [0, 1]
    decay:  (T, 1)  — скорость затухания [0, 1]

    Возвращает context: (T, D)
    context[t] = sum(затухание[i→t] × mass[i] × tokens[i]) для i=0..t
    """
    T, D = tokens.shape
    context = torch.zeros_like(tokens)

    for t in range(T):
        accumulated_decay = 1.0
        for i in range(t, -1, -1):    # идём назад от t до 0
            context[t] += accumulated_decay * mass[i] * tokens[i]
            accumulated_decay *= decay[i].item()  # умножаем на decay этого токена

    return context

Запусти с простыми числами и убедись что результат разумный.

Задание 12.3 — Мысленный эксперимент

Мы сказали что "France" получает высокую mass. Но откуда модель это узнаёт? Ведь mass_proj — это просто умножение на матрицу весов.

Подумай: как в процессе обучения mass_proj учится давать "France" высокий mass? Какой сигнал (градиент) поступает к mass_proj и когда он увеличивает веса?

Подсказка: если mass("France") низкий → context плохо запоминает France → модель не предсказывает "Paris" → loss высокий → градиент говорит "увеличь mass для France".

✅ Итоги главы

HDCMemory — механизм памяти который даёт каждому токену доступ к прошлому:

  

    CODE
    
  

  Mass:   насколько важен этот токен [0..1]  — обучается
Decay:  как быстро забывается [0..1]       — обучается

context[t] = Σ (decay[i→t] × mass[i] × token[i])
             i≤t

Реализация: через log-cumsum → матрица W → matmul
Параметров: всего 8,192 (mass_proj + decay_proj)

Это компактно, эффективно, и концептуально близко к Mamba — только проще.
