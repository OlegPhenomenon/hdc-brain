# Глава 21. Трансформер vs HDC-Brain: детальное сравнение

---

Есть два ресторана на одной улице.

Первый — **"Стандарт"**. Работает 10 лет. Меню проверено, всё вкусно, сотни тысяч довольных клиентов, Michelin отзывы. Шеф-повар делает всё по отработанным рецептам.

Второй — **"Эксперимент"**. Открылся год назад. Меню необычное: некоторые блюда сделаны из других ингредиентов, по-другому нарезаны, с нестандартными соусами. Иногда клиенты удивляются — в хорошем смысле. Иногда что-то не работает. Но это **интересно**.

Трансформер — это "Стандарт". HDC-Brain — это "Эксперимент".

Эта глава — сравнительный анализ меню.

---

### Два пути от токена до предсказания

Сначала посмотрим на архитектуры целиком:

**Стандартный трансформер (GPT-стиль)**:

```
"Привет мир" → токены [12453, 892]
                   ↓
         Token Embedding (nn.Embedding)
       + Position Embedding (learned/sinusoidal)
                   ↓
         Block 1:
           LayerNorm
           Multi-Head Attention (Q×K→scores, softmax, ×V)
           Residual
           LayerNorm
           MLP (expand→GELU→contract)
           Residual
                   ↓
         Block 2 ... Block N (N=32 у SmolLM2)
                   ↓
         LayerNorm
         Output projection (weight-tied)
                   ↓
         Logits → Loss
```

**HDC-Brain v14.1**:

```
"Привет мир" → токены [12453, 892]
                   ↓
         Bipolar Codebook (sign(w) через STE)
       + Cyclic Position Permutation (0 параметров)
                   ↓
         ThoughtLoop — 3 прохода через одни блоки:
           Мысль 1:
             Block1 → Block2 → ... → Block8
             Каждый блок:
               HDCMemory (mass+decay, 8K params)
               LayerNorm + Residual
               HDC Binding Attention (q*k, sigmoid, 12K params)
               LayerNorm + Residual
               ControllerBlock (4096→GELU→4096, 21M params)
           gate_2 → Мысль 2 (снова Block1...Block8)
           gate_3 → Мысль 3 (снова Block1...Block8)
                   ↓
         Output через тот же Codebook.T
                   ↓
         Logits → Loss
```

На первый взгляд похоже. При детальном рассмотрении — принципиально разные подходы к трём ключевым задачам.

---

### Разница №1: Attention — самое радикальное отличие

Это главное место где мы расходимся с трансформером.

**Стандартный Multi-Head Attention** (например, GPT-2 Medium, D=1024, H=16):

```python
# Для каждого слоя нужны 4 матрицы:
W_q = (1024, 1024)  # проекция запросов
W_k = (1024, 1024)  # проекция ключей
W_v = (1024, 1024)  # проекция значений
W_o = (1024, 1024)  # выходная проекция

# Параметров на один слой:
4 × 1024 × 1024 = 4,194,304 ≈ 4M на слой
# Для 24 слоёв:
24 × 4M = 100M параметров только на attention!
```

А у GPT-2 Large (D=1280): ещё больше. У GPT-3 (D=12288, H=96): огромно.

**HDC Binding Attention** у нас (D=4096, H=4):

```python
# Параметры attention для всех 8 блоков вместе:
bv_q = (4, 1024)  # 4 головы × 1024 = 4,096
bv_k = (4, 1024)  # = 4,096
bv_v = (4, 1024)  # = 4,096
# Итого: 12,288 параметров на ВСЕ 8 блоков

# Сравнение с трансформером D=4096, H=32:
# Стандартный 1 слой: 4 × 4096 × 4096 = 67,108,864 ≈ 67M
# Наш весь attention: 12,288

# Разница: 67M / 12K = 5,461 раз!
```

Как это возможно?

В стандартном attention: `Q @ K.T` — это матричное умножение, нужны большие матрицы проекций.

В HDC Binding: `q * k` — поэлементное умножение! Нет T×T матрицы.

```
Стандартный: (B, H, T, D/H) @ (B, H, D/H, T) → (B, H, T, T)
             Размер матрицы растёт как T²

Наш:         (B, H, T, D/H) * (B, H, T, D/H) → (B, H, T, D/H)
             Размер остаётся линейным от T
```

**Что теряем:** трансформер вычисляет попарное сходство **каждого** токена с **каждым** другим. Наш — нет, он делает что-то другое.

**Что получаем:** в 5000 раз меньше параметров в attention. Больше ресурсов для ControllerBlock.

### Разница №2: HDCMemory — явная память которой нет в трансформере

Трансформер хранит информацию о прошлом через attention: при каждом forward pass заново вычисляется "кто на кого смотрит".

```
Трансформер: нет внутренней памяти
  token_5 → attention → смотрит на token_1,2,3,4 → выбирает важное
  (всё пересчитывается каждый раз)
```

У нас есть **HDCMemory** — явная накопленная память:

```
HDCMemory: есть накопленное состояние
  token_1 обработан: context += decay * mass * token_1
  token_2 обработан: context += decay * mass * token_2  (предыдущие затухают)
  ...
  token_5 видит: взвешенную сумму всего прошлого, с decay
```

Это похоже на:

- **Трансформер**: библиотека без каталога — чтобы найти книгу, обходи все полки каждый раз
- **HDCMemory**: библиотека с живым каталогом — актуальный индекс обновляется при каждой новой книге

Технически HDCMemory ближе к **LSTM** (Long Short-Term Memory) чем к трансформеру — только реализован параллельно через log-cumsum, а не через рекуррентные шаги.

### Разница №3: ThoughtLoop — итеративное мышление

Стандартный трансформер: данные проходят через N слоёв ровно один раз.

```
token → Layer1 → Layer2 → ... → Layer32 → prediction
(каждый слой разные параметры, одно направление)
```

Наш ThoughtLoop: 8 блоков применяются 3 раза.

```
token → [Block1...Block8] → мысль 1
                    ↓ (gate_2)
       → [Block1...Block8] → мысль 2
                    ↓ (gate_3)
       → [Block1...Block8] → мысль 3
                    ↓
             prediction
```

Сравнение глубины:

```
SmolLM2-360M: 32 слоя × 1 проход = 32 эффективных шага
HDC-Brain:     8 блоков × 3 прохода = 24 эффективных шага

Параметров в блоках:
SmolLM2: 32 × ~11M = 352M
HDC-Brain: 8 × 21M = 168M  (в 2× меньше за похожую глубину)
```

Это называется **parameter sharing** (разделение параметров). Universal Transformer (2018) предложил ту же идею, мы развили её с learned gates и thought positions.

### Разница №4: Представление токенов

**Стандартный nn.Embedding:**

```python
embedding = nn.Embedding(vocab_size, dim)  # float32
# Веса: любые вещественные числа
# 50257 × 768 = 38.6M параметров (GPT-2 small)
```

**Bipolar Codebook (наш):**

```python
codebook = nn.Embedding(vocab_size, dim)  # обучается как float32
# Но через STE выходит бинарным:
h = sign(codebook.weight)  # {-1, +1}
# 32768 × 4096 = 134M параметров
```

Зачем биполярные векторы вместо вещественных?

1. **Quasi-orthogonality**: в 4096D пространстве 32768 биполярных векторов практически ортогональны друг другу. Каждый токен имеет уникальный "ДНК-отпечаток" — даже похожие слова могут быть почти перпендикулярными.

2. **HDC математика**: операции Bind (×) и Bundle (+) работают именно с биполярными векторами. Permutation (сдвиг) тоже.

3. **Будущая оптимизация**: {-1,+1} можно хранить как биты. 32768 × 4096 бит = 16 МБ вместо 512 МБ. Пока не реализовано, но потенциал есть.

### Разница №5: Позиционное кодирование

| Метод | Параметры | Ограничение длины | Качество |
|-------|-----------|------------------|---------|
| Learned | 1.5M | максимум 2048 | хорошее |
| Sinusoidal | 0 | нет | хорошее |
| RoPE | ~0 | нет | лучшее (стандарт 2024) |
| Cyclic Permute | 0 | нет | неизвестно (наш) |

**Cyclic Permutation:**

```python
# Позиция p → сдвигаем вектор токена на p позиций циклически
shift = p % hdc_dim
indices = (torch.arange(hdc_dim) - shift) % hdc_dim
token_with_pos = token.gather(-1, indices)
```

Каждая позиция — уникальная перестановка вектора. Математически чистое HDC-решение.

Недостаток по сравнению с RoPE: RoPE кодирует **относительные** позиции (расстояние между токенами). Cyclic Permute — **абсолютные** позиции. Для длинного контекста RoPE работает лучше.

### Честное сравнение производительности

Сравниваем с SmolLM2-360M:

| Характеристика | SmolLM2-360M | HDC-Brain v14.1 |
|---|---|---|
| Параметры | 360M | 299M |
| Эффективных слоёв | 32 | 24 |
| Hidden dim | 960 | 4096 |
| Attention params (всего) | ~80M | 12K |
| Токенов обучения | 600 млрд | ~2 млрд |
| BPB на английском | ~2.0 | неизвестно |
| BPB на русском (~40K шагов) | ~3.5 | ~4.2 |

Прямое сравнение нечестное: SmolLM2 обучен в **300 раз дольше**.

Если сравнивать за одинаковое количество обучающих токенов — наши результаты конкурентоспособны.

### Что выиграли и что проиграли

Быть честными с собой важно. Вот честный список:

**Потеряли по сравнению с трансформером:**
```
✗ FlashAttention: несовместим (другой тип attention)
✗ torch.compile: несовместим (Python-цикл в ThoughtLoop)
✗ Выразительность attention: нет попарных T×T сходств
✗ Стандартизация: меньше готовых инструментов и библиотек
✗ Доказанность: трансформер исследован годами, мы — нет
✗ Поддержка сообщества: GPT-архитектуру понимают все
```

**Приобрели:**
```
✓ Явная память: HDCMemory — контекст не пересчитывается заново
✓ Parameter efficiency: 8 блоков = 24 шага (в 3× экономия)
✓ Attention compact: 12K vs 67M (в 5000× меньше)
✓ Позиции бесплатно: 0 параметров, работает на любой длине
✓ HDC математика: быстрый Hamming distance для поиска
✓ Исследовательская новизна: никто так не делал
✓ Возможности будущего: бинарные операции, hardware optimization
```

---

### Что взять из трансформера в будущих версиях

Если делать HDC-Brain v2, стоит подумать о:

**RoPE вместо Cyclic Permutation** — лучше работает на длинных контекстах, легко добавить.

**Развёрнутый ThoughtLoop** — три отдельные функции вместо Python-цикла, тогда torch.compile заработает и даст 2× ускорение.

**Больше голов в attention** — у нас 4, у трансформеров бывает 32-64. Больше голов = модель смотрит на данные с разных "точек зрения".

**Grouped Query Attention (GQA)** — компромисс: меньше KV-голов, много Q-голов. Используется в современных моделях для экономии памяти.

### Итоги главы

HDC-Brain — не "лучше" трансформера. Это другой выбор компромиссов:

```
Attention:
  Трансформер:  Q@K.T, T×T матрица, 67M params/слой
  HDC-Brain:    q*k поэлементно, 12K params всего
  Разница: 5000×

Memory:
  Трансформер:  нет явной памяти, attention пересчитывает
  HDC-Brain:    HDCMemory, weighted decay sum

Глубина:
  Трансформер:  32 независимых слоя
  HDC-Brain:    8 блоков × 3 мысли = 24 эффективных

Токены:
  Трансформер:  float32 embeddings
  HDC-Brain:    {-1,+1} bipolar, quasi-orthogonal

Позиции:
  Трансформер:  RoPE (лучше) или learned
  HDC-Brain:    Cyclic Permute (0 params, проще)

Совместимость:
  Трансформер:  FlashAttention, torch.compile — всё работает
  HDC-Brain:    ни то, ни другое не совместимо
```

Исследовательская ценность в том, что мы пробуем другой путь.
