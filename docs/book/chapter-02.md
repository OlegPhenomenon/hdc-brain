## Лена и 40 000 товаров

Лена — Rails-разработчик в интернет-магазине. К ней приходит тикет:

> "Сделай блок 'вам также понравится'. Показывай похожие товары."

Лена открывает `rails console`:

```ruby
Product.count  # => 40_000
Product.first.name  # => "Nike Air Max 90 кроссовки мужские белые 42р"
```

40 000 товаров. Как объяснить компьютеру что кроссовки Nike и Adidas похожи друг на друга, но оба не похожи на туфли?

Для ActiveRecord "Nike Air Max" и "Adidas Gazelle" — просто строки. `==` вернёт `false`, `LIKE` найдёт только буквальные совпадения. А нужна **похожесть по смыслу**.

Лена думает: "А что если описать каждый товар несколькими числами? Спортивность, формальность, популярность, цена... Тогда похожие товары будут иметь похожие числа."

Она только что придумала **вектор**.

## Запусти: превращаем товары в числа

Создай файл `vectors.py` и запусти:

```python
кроссовки_nike  = [9, 1, 8, 4]
кеды_adidas     = [8, 2, 7, 3]
туфли_oxford    = [1, 9, 4, 7]
кроссовки_asics = [9, 1, 6, 6]

# Каждое число — характеристика:
# [спортивность, формальность, популярность, цена]
#   1=низкая ... 10=высокая

print("Nike:  ", кроссовки_nike)
print("Adidas:", кеды_adidas)
print("Oxford:", туфли_oxford)

# Посмотри на числа глазами:
# Nike [9,1,8,4] и Adidas [8,2,7,3] — почти одинаковые!
# Nike [9,1,8,4] и Oxford [1,9,4,7] — полная противоположность
```

Вот и всё. **Вектор** — это просто список чисел. Каждое число описывает одну характеристику.

Четыре числа — четыре характеристики — **4-мерный вектор**. "Мерный" = "сколько чисел в списке". 4 числа = 4 измерения. Ни больше, ни меньше.

Если бы это был Rails, вектор — это как `attributes.values` у модели. У `Product` есть `:sportiness`, `:formality`, `:popularity`, `:price` — четыре колонки. Значения этих колонок и есть вектор товара.

## Как измерить "похожесть"

У нас есть числа. Но как именно сказать "Nike похож на Adidas больше чем на Oxford"?

### Первая идея: расстояние

Как на карте — чем ближе два города, тем более они "рядом":

```
Nike  = [9, 1, 8, 4]
Adidas = [8, 2, 7, 3]

Расстояние = √((9-8)² + (1-2)² + (8-7)² + (4-3)²)
           = √(1 + 1 + 1 + 1)
           = √4 = 2
```

Работает! Но есть ловушка. Представь два описания одного товара в разных шкалах:

```
версия_А = [9, 1, 8, 4]       (шкала 1..10)
версия_Б = [90, 10, 80, 40]   (шкала 1..100)
```

По расстоянию они далеко друг от друга — хотя описывают одно и то же!

### Вторая идея: направление, а не расстояние

Представь два луча из одной точки. Один указывает "в сторону Nike", другой — "в сторону Adidas":

```
             Nike
              ↗
             /  ← маленький угол → похожи!
            /
───────────●─────────── Adidas
           |
           |
           ↓
       Oxford (совсем в другую сторону)
```

Угол между лучами — мера похожести. Маленький угол = похожие. Большой = разные.

Эту меру называют **косинусное сходство** (cosine similarity). "Косинусное" — потому что считается через косинус угла между векторами. Не нужно знать тригонометрию — просто запомни что на выходе число от -1 до +1:

```
+1.0  → смотрят в одну сторону (одинаковые)
 0.0  → перпендикулярны (ничего общего)
-1.0  → смотрят в противоположные стороны (полные противоположности)
```

И главное преимущество: [9,1,8,4] и [90,10,80,40] дают косинусное сходство **+1.0** — потому что направление одно и то же. Масштаб не важен.

## Запусти: считаем похожесть

Добавь в `vectors.py`:

```python
import math

def cosine_similarity(a, b):
    # Шаг 1: скалярное произведение
    # "Насколько два вектора смотрят в одну сторону"
    # Перемножаем пары элементов и складываем:
    # a[0]*b[0] + a[1]*b[1] + a[2]*b[2] + ...
    dot = sum(x * y for x, y in zip(a, b))

    # Шаг 2: длины векторов
    len_a = math.sqrt(sum(x**2 for x in a))
    len_b = math.sqrt(sum(x**2 for x in b))

    # Шаг 3: делим
    return dot / (len_a * len_b)

print(f"Nike  ↔ Adidas: {cosine_similarity(кроссовки_nike, кеды_adidas):.3f}")
print(f"Nike  ↔ Oxford: {cosine_similarity(кроссовки_nike, туфли_oxford):.3f}")
print(f"Nike  ↔ Asics:  {cosine_similarity(кроссовки_nike, кроссовки_asics):.3f}")
```

Запусти и посмотри. Nike и Adidas будут близки к 1.0. Nike и Oxford — значительно ниже.

Давай разберём что тут происходит по шагам. Возьмём Nike и Adidas:

```
Nike   = [9, 1, 8, 4]
Adidas = [8, 2, 7, 3]

Шаг 1 — скалярное произведение:
  9*8 + 1*2 + 8*7 + 4*3 = 72 + 2 + 56 + 12 = 142

Шаг 2 — длины:
  |Nike|   = √(81 + 1 + 64 + 16) = √162 ≈ 12.7
  |Adidas|  = √(64 + 4 + 49 + 9)  = √126 ≈ 11.2

Шаг 3 — делим:
  142 / (12.7 × 11.2) ≈ 0.997 — почти 1.0! Очень похожи.
```

**Скалярное произведение** (dot product) — название пугает, но суть простая: перемножаешь пары элементов и складываешь. Одно число на выходе. Чем больше оно — тем более "согласованы" два вектора.

## Формула

Ты уже видел код и ручной расчёт. Вот формула — она описывает то же самое:

```
              A · B
sim(A, B) = ─────────
             |A| × |B|
```

Каждый символ:

- `A · B` — скалярное произведение (та самая `sum(x*y for x,y in zip(a,b))`)
- `|A|` — длина вектора A (та самая `sqrt(sum(x**2 for x in a))`)
- Результат — число от -1 до +1

Ты уже знаешь что это делает — считали руками выше. Формула — просто компактная запись того же кода.

## Пространство смысла

До сих пор мы работали с товарами. Но точно так же работает и с **словами**.

Вспомни главу 1: языковая модель предсказывает следующее слово. Чтобы это делать, модель должна "понимать" что слова "кот" и "собака" — похожие (оба домашние животные), а "кот" и "стол" — нет.

Как? Превращаем каждое слово в вектор — список чисел. Похожие слова получают похожие числа.

Если бы слова были 2-мерными (чтобы нарисовать на бумаге):

```
              "живые"
                 │
   волк •        │        • тигр
                 │
  собака •  кот •│
                 │
─────────────────┼───────────────────
  "домашнее"     │        "дикое"
                 │
    стол •       │
  машина •       │ • скала
                 │
             "неживые"
```

Животные — вверху. Вещи — внизу. Домашние — слева. Дикие — справа.

Никто не задавал эти оси вручную. Они **возникают сами** когда модель учится предсказывать следующее слово (как в главе 1). Модель сама обнаруживает что "кот" и "собака" часто появляются в одинаковых контекстах ("домашний питомец спит на..."), и располагает их рядом.

Знаменитый результат 2013 года (Word2Vec):

```
вектор("Король") − вектор("Мужчина") + вектор("Женщина") ≈ вектор("Королева")
```

Направление "мужчина → женщина" совпадает с направлением "Король → Королева". Геометрия смысла.

## Размерность: почему 4096 чисел, а не 4

Вернёмся к главному вопросу. Мы описывали товары 4 числами. Но в настоящих языковых моделях каждое слово описывается **тысячами** чисел. В HDC-Brain — **4096** чисел на одно слово.

Зачем так много? Давай по аналогии.

Представь базу данных людей. Сначала две колонки:

```
Иван = [180 рост, 80 вес]
Пётр = [179 рост, 81 вес]     ← почти неразличимы!
```

Иван и Пётр выглядят одинаковыми — потому что мы описываем их только двумя числами. Добавь колонок:

```
Иван = [180, 80, 35 лет, Москва, программист, рок-музыка, ...]
Пётр = [179, 81, 60 лет, Питер, врач, джаз, ...]
```

Теперь видно что это совершенно разные люди. **Больше чисел = больше "осей сравнения" = точнее различаешь.**

Для слов то же самое. Словарь HDC-Brain содержит **32 768 токенов** (помнишь из главы 1 — модель работает с токенами, кусочками слов). Чтобы каждый из них имел уникальное "место" в пространстве — нужно достаточно осей. С 4 числами все 32 768 токенов слипнутся. С 4096 числами — разместятся свободно, не мешая друг другу.

Это как разница между `text` и `jsonb` в PostgreSQL. `text` — одна колонка, одно сравнение. `jsonb` с 4096 ключами — 4096 колонок, каждый токен описан со всех сторон.

Запомни: **размерность** (dimension) = количество чисел в векторе. В наших примерах — 4. В HDC-Brain — 4096. Чем больше размерность, тем больше информации помещается в вектор.

## Матрица: несколько векторов вместе

Итак, одно слово = один вектор (один список чисел). А что если у нас целое предложение?

Предложение = несколько слов = несколько векторов. Записываем их один под другим, как строки в таблице:

```
Предложение "кот сидит на":

         число_0  число_1  число_2  число_3
кот    [  0.9,   -0.7,     0.9,    -0.8 ]   ← вектор слова "кот"
сидит  [ -0.2,    0.5,     0.1,     0.3 ]   ← вектор слова "сидит"
на     [  0.0,    0.1,    -0.1,     0.0 ]   ← вектор слова "на"

Это таблица 3×4 (3 строки, 4 столбца)
```

Такая таблица чисел называется **матрица** (matrix). Это не что-то новое — просто несколько векторов, сложенных в таблицу. Как список записей в базе данных: каждая строка — один объект, каждый столбец — одна характеристика.

В реальном HDC-Brain размеры больше, но принцип тот же:

```
512 слов × 4096 чисел в каждом = матрица 512×4096
```

512 строк (одна на каждое слово в предложении), в каждой строке 4096 чисел. Всего 2 миллиона чисел — и это только одно предложение!

## Тензор: массив любой формы

Мы уже знаем:
- **Вектор** — список чисел (одна строка). Описывает одно слово.
- **Матрица** — таблица чисел (несколько строк). Описывает одно предложение.

А что если у нас **несколько предложений**? Это будет несколько таблиц, сложенных "в стопку". Для этого есть общее название — **тензор** (tensor).

Тензор — это массив чисел любой формы:
- 1D тензор = вектор (одно слово)
- 2D тензор = матрица (одно предложение)
- 3D тензор = стопка матриц (несколько предложений)

В PyTorch (библиотека для нейросетей на Python) все данные хранятся в тензорах. Это как Ruby-массив, но с суперсилами:
- может быть многомерным (не только `[]`, но и `[[]]`, `[[[]]]`...)
- живёт на GPU (быстрые вычисления)
- умеет автоматически считать градиенты (об этом в главе 4)

Если в Ruby всё — объект (`everything is an object`), то в PyTorch всё — тензор.

```python
import torch

# 1D тензор (вектор) — одно слово, 3 числа
v = torch.tensor([1.0, 2.0, 3.0])
print(v.shape)       # torch.Size([3])

# 2D тензор (матрица) — 3 слова, по 2 числа каждое
m = torch.tensor([[1, 2], [3, 4], [5, 6]])
print(m.shape)       # torch.Size([3, 2]) — 3 строки, 2 столбца

# 3D тензор — 12 предложений, по 512 слов, по 4096 чисел
t = torch.randn(12, 512, 4096)
print(t.shape)       # torch.Size([12, 512, 4096])
```

`.shape` — как `.size` в Ruby, но показывает размер по каждому измерению.

## Батч: обрабатываем пачками

Зачем нам 3D тензор (несколько предложений сразу)? Потому что GPU работает быстрее когда данных много.

Это как `find_in_batches` в Rails:

```ruby
# Медленно: по одному
Product.find_each { |p| process(p) }

# Быстро: пачками по 1000
Product.find_in_batches(batch_size: 1000) { |batch| process(batch) }
```

В нейросетях так же. Подаём не одно предложение, а сразу 16 — это **батч** (batch, "пачка").

Давай соберём всё что мы узнали. Один батч в HDC-Brain выглядит так:

```
16 предложений × 512 токенов в каждом × 4096 чисел на токен

Тензор формы: (16, 512, 4096)
         ↑         ↑       ↑
         |         |       └─ размерность вектора (сколько чисел описывают один токен)
         |         └───────── длина предложения (сколько токенов в одном предложении)
         └─────────────────── размер батча (сколько предложений в пачке)
```

В коде HDC-Brain ты постоянно будешь видеть:

```python
B, T, D = x.shape
# B = Batch  — сколько предложений в пачке     (как batch_size в Rails)
# T = Time   — сколько токенов в предложении   (как длина строки)
# D = Dim    — размерность вектора              (как количество колонок)
```

**B, T, D** — три буквы которые встречаются в каждой следующей главе. Давай закрепим:

- **B** (Batch) = сколько предложений обрабатываем одновременно. Чем больше, тем быстрее GPU работает. Типичное значение: 8, 16, 32.
- **T** (Time) = сколько токенов (слов/кусочков) в одном предложении. "Time" — потому что токены идут по порядку, как события во времени. В HDC-Brain: 512.
- **D** (Dimension) = размерность вектора. Сколько чисел описывают один токен. Мы уже разобрали выше — в HDC-Brain это 4096.

Всякий раз когда ты видишь `(B, T, D)` или `(16, 512, 4096)` — это "пачка предложений, каждое предложение = последовательность токенов, каждый токен = вектор из чисел".

## Матричное умножение: из одного описания — другое

Вернёмся к товарам Лены. У неё 4 характеристики для каждого товара:

```
Nike = [спортивность=9, формальность=1, популярность=8, цена=4]
```

Приходит маркетолог и говорит: "Мне не нужны эти 4 характеристики. Мне нужны 3 другие: **молодёжность**, **премиальность**, **практичность**."

Лена думает: ну ладно, я могу пересчитать. Молодёжность — это в основном спортивность плюс немного популярности. Премиальность — это формальность и цена. Практичность — это популярность минус цена.

Записываем **правила пересчёта**:

```
молодёжность  = 0.8 × спортивность + 0.1 × формальность + 0.5 × популярность + 0.0 × цена
премиальность = 0.0 × спортивность + 0.7 × формальность + 0.1 × популярность + 0.6 × цена
практичность  = 0.3 × спортивность + 0.0 × формальность + 0.4 × популярность - 0.5 × цена
```

Каждое новое число — это **смесь всех старых**, взятых с разными весами. Эти веса (0.8, 0.1, 0.5, 0.0, ...) можно записать в таблицу:

```
                    молодёжн.  премиальн.  практичн.
спортивность  →  [   0.8,       0.0,        0.3   ]
формальность  →  [   0.1,       0.7,        0.0   ]
популярность  →  [   0.5,       0.1,        0.4   ]
цена          →  [   0.0,       0.6,       -0.5   ]

Это матрица 4×3 (4 строки — старые характеристики, 3 столбца — новые)
```

Эта таблица весов — и есть **матрица-линза**. Она говорит: "из каждой старой характеристики возьми столько-то и смешай."

Теперь подставим числа Nike = [9, 1, 8, 4]:

```
молодёжность  = 0.8×9 + 0.1×1 + 0.5×8 + 0.0×4 = 7.2 + 0.1 + 4.0 + 0.0 = 11.3
премиальность = 0.0×9 + 0.7×1 + 0.1×8 + 0.6×4 = 0.0 + 0.7 + 0.8 + 2.4 = 3.9
практичность  = 0.3×9 + 0.0×1 + 0.4×8 - 0.5×4 = 2.7 + 0.0 + 3.2 - 2.0 = 3.9
```

Результат: Nike = [11.3, 3.9, 3.9] — три новые характеристики. Было 4 числа — стало 3.

Вот это и есть **матричное умножение**: взять вектор, пропустить через таблицу весов, получить новый вектор. В PyTorch записывается через `@`:

```python
import torch

# Вектор Nike: 4 характеристики
nike = torch.tensor([[9.0, 1.0, 8.0, 4.0]])

# Матрица-линза: правила пересчёта из 4 характеристик в 3
lens = torch.tensor([
    [0.8, 0.0, 0.3],    # как спортивность влияет на каждую новую
    [0.1, 0.7, 0.0],    # как формальность влияет
    [0.5, 0.1, 0.4],    # как популярность влияет
    [0.0, 0.6, -0.5],   # как цена влияет
])

result = nike @ lens
print(f"Было:  {nike}")           # [9, 1, 8, 4]     — 4 числа
print(f"Стало: {result}")         # [11.3, 3.9, 3.9] — 3 числа
print(f"Форма: {nike.shape} @ {lens.shape} → {result.shape}")
# (1, 4) @ (4, 3) → (1, 3)
```

Обрати внимание на размеры: вход **(1, 4)**, линза **(4, 3)**, результат **(1, 3)**. Внутренние "четвёрки" совпали и "схлопнулись". Осталось (1, 3).

### Зачем это нужно в нейросетях?

Вспомни: каждое слово — это вектор (список чисел). Нейросеть "смотрит" на слово через такую матрицу-линзу и получает **другое описание того же слова**. Как будто переводит с одного "языка характеристик" на другой.

А теперь ключевой момент: **числа в матрице-линзе — это параметры нейросети.** Те самые миллионы чисел из главы 1 ("чуть-чуть подкручиваем параметры чтобы ошибка уменьшилась").

В начале обучения веса случайные — линза кривая, бесполезная. Но после миллионов примеров нейросеть подбирает такие веса, что пересчёт помогает предсказывать следующий токен. Линза становится "умной".

Например: если матрица-линза размером 4096×4096 — это **16 миллионов весов** которые нужно подобрать. Для одного слоя. А слоёв в модели много. Вот откуда берутся "299 миллионов параметров" в HDC-Brain.

---

## HDC-трюк: ×4000 экономии

Матрица-линза размером 4096×4096 — это **16 миллионов чисел**. Для одного слоя. Это дорого: много памяти, медленно считать.

HDC-Brain делает иначе. Вместо матрицы-линзы (таблица 4096×4096) используется просто **один вектор** (список из 4096 чисел). И вместо матричного умножения `@` — **поэлементное умножение** `*`:

```
Стандартно:  вектор @ матрица(4096×4096) → результат     16M параметров
HDC-трюк:    вектор * вектор(4096)       → результат     4096 параметров
```

Поэлементное умножение — это самое простое что можно придумать. Каждый элемент первого списка умножается на соответствующий элемент второго:

```
[0.9, -0.7, 0.9, -0.8]  *  [0.5, 1.0, 2.0, 0.5]  =  [0.45, -0.7, 1.8, -0.4]

Первое число × первое число = первый результат
Второе число × второе число = второй результат
Третье число × третье число = третий результат
И так далее. Каждая пара — независимо.
```

Запусти и почувствуй разницу в скорости:

```python
import torch, time

dim = 4096
x = torch.randn(1, dim)

# Матричное умножение: 16M параметров
W_matrix = torch.randn(dim, dim)

# Поэлементное: 4096 параметров
W_vector = torch.randn(dim)

t0 = time.time()
for _ in range(1000):
    y1 = x @ W_matrix
print(f"Матричное:     {time.time()-t0:.3f}с  (params: {dim*dim:,})")

t0 = time.time()
for _ in range(1000):
    y2 = x * W_vector
print(f"Поэлементное:  {time.time()-t0:.3f}с  (params: {dim:,})")
```

Разница: 16 миллионов параметров vs 4096 параметров. В **4000 раз** меньше.

Что мы теряем? При матричном умножении каждое выходное число зависит от ВСЕХ входных — это "смешивание". При поэлементном — каждое число зависит только от своего номера. Но в 4096-мерном пространстве это оказывается не страшно (почему — разберём в главе 6).

Это поэлементное умножение в мире Hyperdimensional Computing называется **Bind** — одна из трёх главных операций HDC. Она будет основой всей нашей архитектуры.

---

## Давай подведём итоги: что мы теперь знаем

Вот полная цепочка — от начала главы до конца:

1. **Вектор** = список чисел. Описывает один объект (товар, слово, токен). Похожие объекты → похожие числа.

2. **Косинусное сходство** = мера похожести двух векторов. Число от -1 (противоположные) до +1 (одинаковые). Не зависит от масштаба.

3. **Размерность** = сколько чисел в векторе. В наших примерах — 4. В HDC-Brain — 4096. Чем больше, тем точнее описываем объекты.

4. **Матрица** = несколько векторов в таблице. Описывает целое предложение (каждая строка = один токен).

5. **Тензор** = общее название для массива чисел любой формы. Вектор, матрица, 3D-массив — всё это тензоры.

6. **Батч (B, T, D)** = пачка предложений. B = сколько предложений, T = длина каждого, D = размерность вектора. Три буквы которые будут в каждой следующей главе.

7. **Матричное умножение** `@` = "пересчитать через линзу". Мощно, но дорого (миллионы параметров).

8. **Поэлементное умножение** `*` = HDC Bind. Просто, быстро, в 4000 раз дешевле. Главный трюк нашей архитектуры.
