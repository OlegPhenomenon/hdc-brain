# Глава 8. STE — трюк чтобы обучать необучаемое

---

Представь что ты учишь повара делать блюда только из продуктов категории "A" или "B".
Никаких промежуточных вариантов — только чёрное или белое.

Повар учится. Пробует. Получает обратную связь: "слишком кисло", "слишком сладко".
И **на основе этой обратной связи улучшается**.

Но есть одна проблема.

---

### История про светофор и Диму

Дима — светофор.

Не человек, не нейросеть — именно светофор. Он показывает либо **красный**, либо **зелёный**.
Никакого жёлтого, никакого "почти зелёного", никакого "30% зелёного".

```
Дима получает число x (уровень трафика)
Если x > 0 → загорается ЗЕЛЁНЫЙ  (+1)
Если x < 0 → загорается КРАСНЫЙ  (-1)
```

Теперь его нужно обучить. Начальник говорит:

> "Дима, ты сегодня ошибся — нужен был красный, а ты показал зелёный.
> Насколько тебе нужно изменить своё поведение?"

Дима думает:

> "Хм. Я получил `x = 0.7` → показал зелёный.
> Если бы я получил `x = 0.5` → всё равно показал бы зелёный.
> Если бы получил `x = 100` → всё равно зелёный.
> Если бы получил `x = -0.001` → красный!
>
> Получается, моё поведение никак не зависит от точного значения x —
> пока знак не меняется, результат один и тот же.
> Значит... изменение x почти не влияет на мой результат.
> **Градиент = 0.**"

И Дима не может обучиться. Обратная связь до него не доходит.

---

### В чём математическая суть

Функция Димы называется **sign(x)** — функция знака:

```
sign(x) = +1  если x > 0
sign(x) = -1  если x < 0
sign(x) = 0   если x = 0
```

Вот как она выглядит:

```
      +1  ─────────────────────────
          │
     ─────┼─────────────────────── x
          │
      -1  ─────────────────────────
```

Резкий скачок в нуле. И именно из-за этого скачка — производная везде равна нулю:

```
d/dx sign(x) = 0  при x ≠ 0
               ∞  при x = 0  (технически не определена)
```

Вспоминаем главу 4: обучение нейросети — это **градиентный спуск**.
Нам нужна производная чтобы понять "в какую сторону двигать веса".

Если производная = 0 → градиент не течёт → веса не обновляются → **обучение остановилось**.

---

### Почему это проблема для нас

В главе 7 мы узнали: HDC-Brain использует **bipolar векторы {-1, +1}**.
Это значит, что в процессе работы мы берём обычные числа (веса нейросети) и бинаризуем их через sign().

```
Кодбук хранит:  [-0.37,  0.82, -0.05,  0.61, ...]
После sign():   [  -1,    +1,    -1,    +1,  ...]
```

Это хорошо для скорости. Но как **обучать** кодбук, если gradient через sign() = 0?

Без обучения кодбук будет случайным навсегда. Модель не сможет учиться.

---

### А это называется: Straight-Through Estimator (STE)

В 2013 году Ёша Бенжио (один из трёх "крёстных отцов" глубокого обучения) опубликовал
простейшее решение.

Идея такая:

> **На прямом ходу** (forward pass, когда считаем результат) — используем настоящую бинаризацию.
> **На обратном ходу** (backward pass, когда считаем градиент) — **притворяемся**, что функция была identity.

"Identity" — это математический термин для функции `f(x) = x`.
Её производная = 1. Всегда. Везде.

```
Forward:  y = sign(x)          ← настоящая, честная бинаризация
Backward: dy/dx = 1            ← ЛОЖЬ! Но полезная ложь
```

Возвращаемся к Диме.

Дима-с-STE думает по-другому:

> "Я покажу красный или зелёный честно.
> Но когда начальник спросит `насколько сильно изменение x влияло на мой результат?` —
> я отвечу: `прямо пропорционально`. То есть как будто я просто пропустил x без изменений."

Это ложь? Да. Точная ложь? Нет. Но Дима теперь **обучается**.

Градиент "проходит сквозь" бинаризацию как через прозрачную трубу:

```
Без STE:
  Loss ──→ ... ──→ sign(x) ──→ 0 (стоп!)
                                ↑
                        градиент заблокирован

С STE:
  Loss ──→ ... ──→ sign(x) ──→ продолжается ──→ x.grad = upstream_grad × 1
                                                            ↑
                                              "как будто sign не было"
```

---

### Как это выглядит в коде

В PyTorch есть специальный механизм для создания операций с кастомной производной:

```python
import torch

class STESign(torch.autograd.Function):
    @staticmethod
    def forward(ctx, x):
        # Честная бинаризация
        return x.sign()

    @staticmethod
    def backward(ctx, grad_output):
        # Притворяемся что f(x) = x → производная = 1
        # Просто пропускаем градиент дальше без изменений
        return grad_output  # ← "прозрачная труба"

ste_sign = STESign.apply
```

Проверяем:

```python
w = torch.tensor([0.7, -0.3, 0.1, -0.9], requires_grad=True)

# Forward: честная бинаризация
out = ste_sign(w)
print(f"Вход:    {w.tolist()}")
print(f"Выход:   {out.tolist()}")
# Вход:    [0.7, -0.3, 0.1, -0.9]
# Выход:   [1.0, -1.0, 1.0, -1.0]  ← настоящие bipolar числа!

# Backward: градиент проходит как будто sign не было
loss = out.sum()
loss.backward()
print(f"Градиент: {w.grad.tolist()}")
# Градиент: [1.0, 1.0, 1.0, 1.0]  ← прошёл! не ноль!
```

### Более компактная запись: трюк с detach()

Наша модель использует чуть другую запись — без явного `autograd.Function`.

```python
def ste_binarize(w):
    hard = torch.sign(w)          # бинарный результат
    return (hard - w).detach() + w  # ← вся магия здесь
```

Разберём последнюю строку буквально:

```python
result = (hard - w).detach() + w
```

**Что это значит:**

`.detach()` говорит PyTorch: "не считай градиент через эту часть".

Значение при forward:
```
result = (hard - w) + w = hard   ← получаем бинарный!
```

Градиент при backward:
```
d(result)/dw = d((hard - w).detach())/dw  +  d(w)/dw
             =        0         (detach)  +     1
             =                              1   ← проходит!
```

Один трюк делает сразу две вещи:
- forward выдаёт честный бинарный результат
- backward притворяется что бинаризации не было

Посмотрим на числа:

```
Допустим w = 0.7:

Forward:
  hard = sign(0.7) = 1.0
  result = (1.0 - 0.7) + 0.7 = 0.3 + 0.7 = 1.0  ✓ бинарный

Backward (допустим upstream_grad = -2.0):
  d(result)/dw = 1
  w.grad += upstream_grad × 1 = -2.0  ← обновляемся!
```

### Alpha-scaling: улучшение из статьи XNOR-Net

В нашем коде есть один бонус сверх обычного STE.

Проблема: если вектор `[100, -200, 50, -300]`, а мы бинаризуем в `[+1, -1, +1, -1]` — мы **теряем масштаб**.

Решение из статьи XNOR-Net (2016): сохраняем средний масштаб и умножаем на него:

```python
def _ste_sign(self, w):
    # Считаем "масштаб" вектора
    alpha = torch.mean(torch.abs(w), dim=-1, keepdim=True)
    # alpha = среднее из |w_i| по измерению D

    # Бинаризуем, но сохраняем масштаб
    hard = alpha * torch.sign(w)
    # Вместо [+1, -1, +1] получаем [+alpha, -alpha, +alpha]

    # Обрабатываем нули (знак нуля = 0, нам нужно ±alpha)
    hard = torch.where(hard == 0, alpha * torch.ones_like(hard), hard)

    # STE
    return (hard - w).detach() + w
```

На примере:

```
w = [0.3, -0.8, 0.1, 0.5]

alpha = mean(|0.3|, |-0.8|, |0.1|, |0.5|) = mean(0.3, 0.8, 0.1, 0.5) = 0.425

hard = 0.425 × sign([0.3, -0.8, 0.1, 0.5])
     = 0.425 × [+1, -1, +1, +1]
     = [+0.425, -0.425, +0.425, +0.425]
```

Почему это лучше? Сравним:

```
Без alpha:  знак    = [+1,    -1,    +1,    +1]
С alpha:    hard    = [+0.425, -0.425, +0.425, +0.425]
Оригинал:           = [+0.300, -0.800, +0.100, +0.500]

Без alpha: потеряли всю информацию о масштабе
С alpha: сохранили "средний масштаб" — ближе к оригиналу по норме
```

### Полный код _ste_encode из HDC-Brain

Теперь разберём весь `_ste_encode` построчно:

```python
def _ste_encode(self, idx):
    """STE bipolar encoding.
    Forward:  возвращает бинарные векторы ±alpha
    Backward: градиент проходит к кодбуку
    """

    # === Шаг 1: Берём непрерывные векторы из кодбука ===
    real = self.codebook[idx]
    # idx: (B, T) — индексы токенов
    # real: (B, T, D) — непрерывные float числа, например [-0.37, 0.82, -0.05, ...]

    # === Шаг 2: Считаем масштаб (alpha) ===
    alpha = torch.mean(torch.abs(real), dim=-1, keepdim=True)
    # alpha: (B, T, 1) — одно число на каждый вектор

    # === Шаг 3: Бинаризуем с сохранением масштаба ===
    hard = alpha * torch.sign(real)
    # hard: (B, T, D) — только значения ±alpha

    # === Шаг 4: Обрабатываем нули (sign(0) = 0, а нам нужно ±alpha) ===
    hard = torch.where(hard == 0, alpha * torch.ones_like(hard), hard)

    # === Шаг 5: STE трюк ===
    return (hard - real).detach() + real
    #
    # Forward:  result = (hard - real) + real = hard  ← бинарный
    # Backward: d/d(real) = 0 (detach) + 1 = 1       ← градиент проходит
```

---

### Что происходит при обучении

Следим за одним весом во время тренировки:

```
Шаг 0:  w = +0.10  →  hard = +1  →  loss высокий
        gradient = -0.5 (нужно уменьшить)
        w_new = 0.10 - lr × 0.5 = 0.10 - 0.05 = 0.05

Шаг 1:  w = +0.05  →  hard = +1  →  loss высокий
        gradient = -0.5
        w_new = 0.05 - 0.05 = 0.00

Шаг 2:  w = +0.00  →  hard = 0 → заменяем на +alpha  ← обрабатываем ноль
        gradient = -0.5
        w_new = 0.00 - 0.05 = -0.05

Шаг 3:  w = -0.05  →  hard = -1  ← ЗНАК ПОМЕНЯЛСЯ!
        loss может стать ниже...
        gradient = +0.2 (теперь нужно увеличить)
        w_new = -0.05 + 0.02 = -0.03
```

Вес "прощупывает" пространство вокруг нуля и находит правильную сторону.
В больших моделях (тысячи весов) это работает очень хорошо статистически.

---

### Почему это работает, если это ложь?

Законный вопрос. Мы передаём **неправильный** градиент — так почему модель обучается?

**Причина 1: Направление важнее точности**

Нам нужно знать — увеличивать вес или уменьшать?
STE передаёт правильное **направление** (знак градиента), даже если неточно передаёт **величину**.

```
Правильный градиент: 0     (совсем бесполезен!)
STE градиент:       -0.7   (говорит "уменьши вес")

Даже "неправильный" -0.7 лучше чем правильный 0!
```

**Причина 2: Эмпирия за нас**

За десять лет существования STE — тысячи статей, сотни архитектур.
Все работают. BitNet от Microsoft обучает 7B-параметровую модель с 1-bit весами.

**Причина 3: Суррогатный градиент**

Теоретически (Yin et al., 2019): STE — это оптимизация "непрерывной релаксации" задачи.
Мы обучаем непрерывный вес `w`, а используем бинарный `sign(w)`.
Это как тренироваться на симуляторе, а выступать на настоящих соревнованиях.

---

### STE в истории нейросетей

STE — не единственная вещь такого рода, но самая популярная:

```
BinaryConnect (2015)  — первые бинарные веса с STE, работает!
BinaryNet (2016)      — бинарные веса И активации, 7x быстрее
XNOR-Net (2016)       — XNOR+POPCNT inference, 58x ускорение на CPU
BitNet (2023, MS)     — 1-bit LLM, конкурирует с float16 при 7B параметров
BitNet b1.58 (2024)   — {-1, 0, +1}, ещё мощнее чем чистый {-1, +1}
```

Наша архитектура использует alpha-scaled STE как в XNOR-Net — это проверенный подход.

---

### Формула STE (на самом деле простая)

Запись через "прямой проход" и "обратный проход":

```
Forward:   y = sign(x)
Backward:  ∂L/∂x = ∂L/∂y  (как будто y = x)
```

Или через "остановку градиента":

```
y = (sign(x) - x).stop_gradient + x
  = sign(x) - x  +  x      ← значение
  =       sign(x)           ← forward: честная бинаризация

∂y/∂x = 0  +  1 = 1        ← backward: как identity
```

Вот что каждый символ означает буквально:
- `sign(x)` — берём знак числа: + превращается в +1, − в -1
- `.stop_gradient` / `.detach()` — говорим PyTorch: "не считай производную через это"
- `+ x` в конце — "добавляем x обратно, чтобы его производная (= 1) работала"

### Итоги главы

STE — трюк позволяющий обучать бинарные веса:

```
Проблема:  sign(x) → производная = 0 → нет обучения

Решение:   В forward  → используем настоящий sign(x)
           В backward → притворяемся что функция = identity → производная = 1

Код:  (sign(w) - w).detach() + w
      ↑                         ↑
    значение = sign(w)     производная = 1
```

В HDC-Brain STE применяется дважды:
- `_ste_encode` — бинаризует кодбук при кодировании токенов
- `_ste_sign` — бинаризует binding vectors в MultiHeadBindingAttention

Почему работает несмотря на "ложь":
- Направление градиента **верное** → обновляем веса в правильную сторону
- Подтверждено тысячами работ: BinaryConnect, XNOR-Net, BitNet
- Альтернатива (настоящий градиент = 0) хуже любой лжи
