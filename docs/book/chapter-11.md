# Глава 11. Codebook — словарь как отпечатки пальцев

---

В полицейском управлении есть база отпечатков пальцев.

Каждый человек в базе — уникальный набор завитков и линий.
Берёшь отпечаток с места преступления — сверяешь с базой — находишь совпадение.

Codebook — это точно то же самое. Только вместо отпечатков — векторы.
Вместо людей — слова (точнее, токены).

Каждый токен имеет свой уникальный "отпечаток" — вектор из 4096 чисел.
Берёшь ID токена — ищешь в таблице — получаешь вектор.

---

### От слова к вектору

В 2013 году Google выпустил Word2Vec. Это была маленькая революция:

```
"king" - "man" + "woman" ≈ "queen"
"Paris" - "France" + "Germany" ≈ "Berlin"
```

Оказалось, что если правильно обучить нейросеть на текстах,
слова с похожим значением получают похожие векторы.
"Кошка" и "собака" — рядом в пространстве.
"Кошка" и "банк" — далеко.

Это называется **эмбеддинг** (embedding) — вложение слова в числовое пространство.

В современных языковых моделях эмбеддинг-таблица обучается **прямо вместе с моделью**.
Не нужно сначала учить Word2Vec отдельно — всё учится в одной тренировке.

---

### Устройство кодбука

В PyTorch таблица эмбеддингов называется `nn.Embedding`:

```python
import torch
import torch.nn as nn

# Таблица: 32,000 строк, каждая строка = вектор из 4096 чисел
codebook = nn.Embedding(32000, 4096)

# Размер в памяти: 32,000 × 4,096 × 4 байта = 524 МБ
```

Как пользоваться:

```python
# Получаем вектор для одного токена
token_id = 4217         # например, токен "capital"
vector = codebook(torch.tensor([token_id]))
print(vector.shape)     # (1, 4096)
print(vector[0, :5])    # первые 5 чисел: tensor([-0.014,  0.019, -0.007,  0.023, -0.011])

# Получаем векторы для целого предложения (батча токенов)
ids = torch.tensor([[4217, 310, 6355]])  # "capital of France"
vectors = codebook(ids)
print(vectors.shape)    # (1, 3, 4096)  — батч × длина × размерность
```

Это просто **индексация в матрицу**. Подаёшь номер строки — получаешь строку.

### Наш кодбук: с STE

В HDC-Brain кодбук устроен чуть сложнее:

```python
# Храним в виде обычных float числе (для обучения)
self.codebook = nn.Parameter(torch.randn(vocab_size, hdc_dim) * 0.02)
```

При использовании — пропускаем через STE (глава 8):

```python
def _ste_encode(self, idx):
    # Берём float числа из таблицы
    real = self.codebook[idx]         # (B, T, D) — обычные числа

    # Считаем масштаб
    alpha = torch.mean(torch.abs(real), dim=-1, keepdim=True)

    # Бинаризуем: только ±alpha
    hard = alpha * torch.sign(real)
    hard = torch.where(hard == 0, alpha * torch.ones_like(hard), hard)

    # STE: forward=бинарный, backward=градиент проходит
    return (hard - real).detach() + real
```

Хранится: float32 (обычные числа, для обучения).
Используется: bipolar {±alpha} (быстрые бинарные операции).

Разница с обычным `nn.Embedding` — только в последнем шаге бинаризации.

### Сколько параметров съедает кодбук

Посчитаем честно:

```
vocab_size = 32,000 токенов
hdc_dim    = 4,096  чисел на токен

Параметров = 32,000 × 4,096 = 131,072,000 ≈ 131 миллион
```

В памяти при хранении (float32):
```
131,072,000 × 4 байта = 524 МБ
```

В памяти при инференсе (bipolar):
```
131,072,000 × 1 бит = 16 МБ  (в 32 раза меньше!)
```

И это **43.8% всей модели** (из 299M параметров).
Почти половина модели — просто таблица "токен → вектор".

Почему так много? Потому что `hdc_dim = 4096` намного больше чем у маленьких моделей:

```
SmolLM2:   49,152 токенов × 960  размерность = 47M параметров  (13% модели)
HDC-Brain: 32,000 токенов × 4096 размерность = 131M параметров (44% модели)
```

Это цена высокой размерности HDC. Зато в этом пространстве векторы почти ортогональны (глава 5).

### Трюк с весовым связыванием (Weight Tying)

Вот один из красивейших трюков в нейросетях.

Стандартная языковая модель имеет **две** таблицы:

```
1. Input embedding:   токен → вектор        (на входе)
2. Output projection: вектор → токен        (на выходе)

Размер каждой: vocab_size × dim
Итого: 2 × 131M = 262M параметров
```

Но подумаем: что делает input embedding?
— "Переводит" ID токена в вектор.

Что делает output projection?
— "Переводит" вектор обратно в ID токена (через сравнение похожести).

Это **симметричные операции**. Одним словарём!

Поэтому мы используем **одну таблицу** для обоих направлений:

```python
# На входе: берём строку из таблицы
x = self._ste_encode(idx)           # codebook[idx] → бинарный вектор

# На выходе: умножаем на ту же таблицу (транспонированную)
logits = F.linear(h, self.codebook) * self.output_scale
#                      ↑
#             та же таблица self.codebook!
#             F.linear(h, W) вычисляет h @ W^T
```

Экономия: 131 миллион параметров.

Аналогия: двуязычный словарь. Русско-английский и английско-русский — одна книга.
Если хочешь "перевести" слово из одного языка — ищешь слева направо.
Если наоборот — справа налево. Та же книга, разное направление чтения.

---

### Почему начальные веса × 0.02?

```python
self.codebook = nn.Parameter(torch.randn(vocab_size, hdc_dim) * 0.02)
```

`torch.randn` даёт случайные числа из N(0, 1) — среднее ноль, стандартное отклонение 1.

Умножаем на 0.02 → получаем числа примерно от -0.06 до +0.06.

Зачем такие маленькие?

Представь что начальные числа большие, например от -10 до +10:

```
После STE:
  alpha ≈ mean(|веса|) ≈ 5-7
  hard = ±5 ... ±7

На выходе модели:
  logits = h @ codebook^T
  Скалярное произведение 4096 чисел ≈ ±5 × 4096 = огромные числа!

После softmax(1000) vs softmax(-1000):
  Всё вероятность уходит в один токен, градиент исчезает → обучение сломано
```

Маленькие начальные веса:

```
alpha ≈ 0.016
hard = ±0.016

logits ≈ небольшие числа → нормальный softmax → нормальное обучение
```

`0.02` — стандартное значение для инициализации трансформеров (из статьи GPT-2).

---

### output_scale: масштабирование выхода

```python
self.output_scale = nn.Parameter(torch.tensor(hdc_dim ** -0.5))
# hdc_dim = 4096
# hdc_dim ** -0.5 = 1 / √4096 = 1/64 ≈ 0.0156
```

Зачем делить на корень из размерности?

Представь два случайных вектора `h` и `codebook[i]` длиной D.
Их скалярное произведение (dot product) растёт с размерностью:

```
E[h · codebook[i]] = 0        (среднее = 0, случайные векторы)
Var[h · codebook[i]] ≈ D      (дисперсия растёт как D)
Std[h · codebook[i]] ≈ √D     (стандартное отклонение = √D)
```

В нашем случае D = 4096, √D = 64.

Без масштабирования логиты были бы в 64 раза больше нормы.
Softmax бы выбирал один токен с почти 100% вероятностью.
Это называется "collapsed distribution" — плохо для обучения.

Деление на √D нормализует: логиты снова в нормальном диапазоне.

Интересный момент: `output_scale` — **обучаемый параметр** (`nn.Parameter`).
Начинается с 1/√D, но может меняться в процессе обучения.
Модель сама решает насколько масштабировать свои предсказания.

### Полный путь токена через кодбук

Смотрим на то что происходит шаг за шагом:

```
Ввод: токен "capital" → ID = 4217

Шаг 1: Индексация в кодбук
  real = codebook[4217]
  real: вектор 4096 чисел, например [-0.014, 0.019, -0.007, 0.023, ...]
  (в начале обучения — случайные маленькие числа)
  (после обучения — числа отражают смысл слова "capital")

Шаг 2: Бинаризация (STE)
  alpha = mean(|real|) ≈ 0.015
  hard = ±0.015 в зависимости от знака каждого числа
  result = (hard - real).detach() + real  → возвращаем hard

Шаг 3: Позиционное кодирование (циклический сдвиг)
  Если "capital" стоит на позиции 3:
  shifted = roll(hard, shift=-3)
  Теперь вектор "знает" что он на 3-й позиции

Шаг 4: Поступает в первый HDCBlock
  Обработка начинается...
```

### Что кодбук выучивает в процессе обучения

В начале тренировки все токены имеют случайные, несвязанные между собой векторы.

После обучения токены с похожим **контекстом** (словами вокруг) сближаются в пространстве:

```
"Paris", "London", "Berlin", "Tokyo"  →  близко друг к другу (столицы)
"cat", "dog", "bird", "fish"          →  в своём кластере (животные)
"the", "a", "an"                      →  рядом (артикли)
"run", "runs", "ran", "running"       →  похожие (формы одного глагола)
```

Проверить это можно просто — найти ближайших соседей в кодбуке по косинусному сходству:

```python
import torch.nn.functional as F

def find_nearest(model, sp, word, top_k=5):
    """Найти токены с похожими векторами."""
    # Получаем ID слова
    word_id = sp.encode(word)[0]
    word_vec = model.codebook[word_id]  # вектор слова

    # Считаем сходство со всем словарём
    all_vecs = model.codebook
    similarities = F.cosine_similarity(word_vec.unsqueeze(0), all_vecs, dim=-1)

    # Берём топ-k (пропускаем само слово)
    top_ids = similarities.topk(top_k + 1).indices[1:]

    # Переводим ID обратно в слова
    return [sp.decode([i.item()]) for i in top_ids]

# Пример после обучения:
print(find_nearest(model, sp, "Paris"))
# Ожидаем: ["London", "Berlin", "Rome", "Madrid", "Vienna"]
```

### Формула (на самом деле просто индексация)

```
Codebook: матрица E ∈ R^(V × D)
  V = vocab_size = 32,000
  D = hdc_dim = 4,096

Для входа: x = E[token_id]        ← берём строку
Для выхода: logits = h × E^T      ← умножаем на транспонированную
```

Вот что каждый символ означает буквально:
- `E` — таблица, содержащая D-мерный вектор для каждого из V токенов
- `E[token_id]` — просто взять строку с номером token_id
- `h × E^T` — для каждого токена посчитать насколько вектор `h` похож на вектор токена
- `×E^T` транспонируем потому что хотим `(1×D) × (D×V) = (1×V)` — один балл на каждый токен

### Итоги главы

Кодбук — это таблица "отпечатков" для каждого токена:

```
Структура: матрица 32,000 × 4,096 = 131M параметров
Хранение:  float32 (обычные числа, обучаются через backprop)
Использование: bipolar через STE (быстрые операции)
```

Ключевые трюки:
- **STE**: float32 хранение, bipolar использование
- **Weight Tying**: одна таблица для входа и выхода → экономим 131M параметров
- **Инициализация × 0.02**: стабильный старт обучения
- **output_scale = 1/√D**: нормализация дисперсии логитов

После обучения похожие токены стоят рядом в пространстве — автоматически.
