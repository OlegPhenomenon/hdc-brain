Но каждое движение требует **разной степени коррекции**! Запястье нужно исправить на 30°, давление — на 2%, плечо — на 0.1°.

Если делать всё на одинаковый шаг — половина движений перекорректирована, половина едва сдвинулась.

Нейронная сеть обучается точно так же. И **градиентный спуск** — это как учитель, который говорит "подправь всё на одну и ту же величину". Работает, но плохо.

### Адаптивный тренер: история Юли и Паши

Юля тренирует теннисистов. У неё двести учеников.

Некоторые делают маленькие ошибки — чуть-чуть неправильно держат ракетку. Им нужна **крошечная коррекция**.

Другие делают огромные ошибки — всё тело разворачивается не туда. Им нужна **большая коррекция**.

Паша — другой тренер — всем говорит одно и то же: "Исправь на 3 сантиметра". Тем кому нужно исправить на 1 миллиметр — перекорректированы. Тем кому нужно 30 сантиметров — почти не помогло.

Юля запоминает **историю каждого ученика**. Если ученик постоянно делает мелкие ошибки — значит он уже почти правильно двигается, не нужна большая коррекция. Если постоянно большие ошибки — нужен смелый шаг.

Это и есть идея **Adam** (Adaptive Moment Estimation). Для каждого параметра — своя история, своя адаптивная коррекция.

### Как Adam запоминает историю

Adam хранит для каждого параметра два числа: `m` и `v`.

```python
# Перед началом обучения:
m = 0  # "средний градиент" — куда в среднем нужно двигаться
v = 0  # "дисперсия градиентов" — насколько сильно прыгают шаги
```

`m` — это как средняя ошибка ученика за последние тренировки.
`v` — это разброс этих ошибок.

На каждом шаге:

```python
g = gradient  # градиент на этом шаге (ошибка сегодня)

# Обновляем "память":
m = 0.9 * m + 0.1 * g       # средняя ошибка (90% старое + 10% новое)
v = 0.95 * v + 0.05 * g * g # разброс ошибок

# Поправка: в начале m и v близки к 0, нужно компенсировать
m_hat = m / (1 - 0.9**step)
v_hat = v / (1 - 0.95**step)

# Шаг обновления:
param -= lr * m_hat / (sqrt(v_hat) + 1e-8)
```

Разберём что делает `sqrt(v_hat)` в знаменателе:

- Если параметр **часто получает большие градиенты** → `v_hat` большой → делим на большое число → **шаг маленький**
- Если параметр **редко обновляется, с маленькими градиентами** → `v_hat` маленький → делим на маленькое число → **шаг большой**

Юля говорит ученику который делает мелкие ошибки: "Не трогай, ты почти правильно". Ученику с большими ошибками: "Смелее, исправь сильнее".

Адаптация произошла автоматически, без ручной настройки!

### AdamW: "правильный" штраф за большие веса

Теперь другая проблема.

Нейронная сеть без ограничений любит **наращивать веса**. Если какой-то вес даёт хороший результат при значении 0.5, модель пробует 1.0, потом 2.0, потом 100.0...

Огромные веса — признак **переобучения**: модель запомнила тренировочные данные, но не научилась обобщать.

Простое решение: штрафуй за большие веса. Это называется **weight decay** (затухание весов).

```python
# Наивная идея (L2-регуляризация внутри Adam):
param -= lr * (gradient + weight_decay * param)
# Проблема: weight_decay попадает в m и v → делится на sqrt(v)
# Это меняет смысл штрафа непредсказуемо

# AdamW (Loshchilov & Hutter, 2017) — правильный способ:
param = param * (1 - lr * weight_decay)  # ← сначала уменьшаем вес
param -= lr * m_hat / (sqrt(v_hat) + eps)  # ← потом обновляем по градиенту
```

Разница: в **обычном Adam** штраф за большие веса "смешивается" с адаптацией и работает странно.
В **AdamW** штраф применяется отдельно, до обновления — чисто и предсказуемо.

"W" в названии — это "Weight decay (correct)".

```python
optimizer = torch.optim.AdamW(
    params,
    lr=3e-4,           # скорость обучения
    betas=(0.9, 0.95), # beta1=0.9 для m, beta2=0.95 для v
    weight_decay=0.1   # 10% уменьшение весов на каждом шаге (после учёта LR)
)
```

Что означает каждый параметр:
- `lr=3e-4` — максимальный шаг обновления (3 × 10⁻⁴ = 0.0003)
- `betas=(0.9, 0.95)` — насколько "помнить" прошлые градиенты (90%) и их разброс (95%)
- `weight_decay=0.1` — насколько штрафовать за большие веса

### Два класса параметров: кодбук особенный

Теперь важный момент нашей архитектуры.

У нас **не один** оптимизатор для всех параметров. У нас **два разных режима**:

```python
# Из train.py:
codebook_params = [p for n, p in model.named_parameters()
                   if 'codebook' in n]     # ← параметры кодбука
other_params    = [p for n, p in model.named_parameters()
                   if 'codebook' not in n]  # ← всё остальное

optimizer = torch.optim.AdamW([
    {'params': other_params,    'lr': 3e-4, 'weight_decay': 0.1},
    {'params': codebook_params, 'lr': 3e-5, 'weight_decay': 0.0},
    #                                  ↑                     ↑
    #                          в 10 раз меньше!          нет штрафа!
], betas=(0.9, 0.95))
```

**Почему кодбук учится в 10 раз медленнее?**

Вспомни главу 8: кодбук — это биполярные векторы `{-1, +1}`, обученные через **STE** (Straight-Through Estimator). STE — это трюк: "делаем вид что `sign()` имеет градиент 1".

Этот трюк хорошо работает только когда реальный вес `w` близко к двоичному `sign(w)`.

Если LR большой → вес прыгает далеко → `w` сильно отличается от `sign(w)` → STE даёт неточные градиенты → кодбук ломается.

Маленький LR = маленькие прыжки = STE остаётся точным приближением = кодбук учится нормально.

Нашли это **экспериментально**: при LR=3e-4 кодбук не сходился. Уменьшили до 3e-5 — работает.

**Почему для кодбука weight_decay=0.0?**

Биполярные векторы: после STE веса `w` превращаются в `sign(w) ∈ {-1, +1}`.

Что делает weight_decay? Уменьшает веса к нулю: `w = w * (1 - 0.1 * 0.0003) ≈ 0.99997 * w`.

Для кодбука это катастрофа: через тысячи шагов `w → 0`. Но `sign(0) = ?` Нет знака у нуля!

Кодбук потеряет всю информацию. Поэтому weight_decay=0.0 — отключаем штраф для биполярных векторов.

### Скорость обучения со временем

Пока мы говорили о **постоянном** learning rate. Но постоянный LR — плохая идея.

История Кости-марафонца:

Костя бежит марафон (42 км).

Первые шаги — он разминается. Нельзя сразу рвануть на полной скорости, мышцы не готовы.

Основная дистанция — Костя бежит быстро, на полную мощь.

Последние километры — он чуть замедляется, бережёт силы, финишная прямая точная.

С постоянной скоростью так не работает: сразу полная — травма. Всегда медленно — не финишируешь вовремя.

**Расписание скорости обучения** (Learning Rate Schedule) — то же самое для нейросети.

### Cosine Schedule: плавное замедление

Самая популярная схема — **косинусное затухание** (cosine decay):

```
LR
│
3e-4 ─╮
│      ╲
│       ╲___
│           ╲___
│               ╲____
│                    ╲___
1e-5 ────────────────────── шаг
0   100   ...    40,000
│    │              │
warmup  обучение   финал
```

Код:

```python
import math

def cosine_lr(step, total_steps, max_lr, min_lr, warmup_steps):
    if step < warmup_steps:
        # Фаза 1: линейный разгон от ~0 до max_lr
        return max_lr * step / warmup_steps

    # Фаза 2: косинусное снижение от max_lr до min_lr
    progress = (step - warmup_steps) / (total_steps - warmup_steps)
    # progress идёт от 0.0 до 1.0

    cosine_factor = 0.5 * (1 + math.cos(math.pi * progress))
    # cos(0) = 1.0 → в начале
    # cos(π) = -1.0 → в конце → (1 + (-1))/2 = 0.0
    # Итого: cosine_factor от 1.0 до 0.0

    return min_lr + (max_lr - min_lr) * cosine_factor
```

Что делает каждая строчка:

1. `progress` — сколько мы прошли от 0 до 1 (0 = начало, 1 = конец)
2. `math.cos(math.pi * progress)` — косинус идёт от cos(0)=1 до cos(π)=-1
3. `0.5 * (1 + ...)` — превращает диапазон [-1..1] в [0..1]
4. `min_lr + (max_lr - min_lr) * cosine_factor` — интерполяция между min и max

В нашем проекте:
```python
WARMUP_STEPS = 100    # ~0.25% от 40K шагов
MAX_LR       = 3e-4   # пик скорости
MIN_LR       = 1e-5   # финальная скорость (в 30 раз меньше пика!)
```

---

### Почему разогрев важен: история нестабильного начала

Первые 100 шагов LR растёт от почти нуля до 3e-4. Это **warmup** (прогрев).

Зачем?

В начале обучения модель инициализирована **случайно**. Все веса — маленькие случайные числа. Градиенты в таком состоянии — непредсказуемы и потенциально огромны.

```
Шаг 1 без warmup:
  Градиент = 15.0 (огромный!)
  LR = 3e-4
  Шаг обновления = 0.0045

Шаг 1 с warmup:
  Градиент = 15.0
  LR = 3e-4 * 1/100 = 3e-6  (в 100 раз меньше!)
  Шаг обновления = 0.000045  ← безопасно
```

Без warmup при старте с большого LR нейросеть может улететь в плохую зону пространства потерь, из которой уже не выберется.

С warmup первые 100 шагов — "нащупываем" правильное направление, потом газуем.

### Страховочный трос: gradient clipping

Даже при правильном LR иногда случается **взрыв градиента**.

Представь: модель идёт по горной тропе. Большинство шагов — нормальные. Но иногда нога попадает в яму — резкий провал и огромный шаг в неожиданном направлении.

`gradient clipping` (обрезание градиентов) — это страховочный трос:

```python
# Из train.py:
grad_norm = torch.nn.utils.clip_grad_norm_(
    model.parameters(),
    max_norm=1.0
)
```

Что происходит внутри:

```python
# Шаг 1: считаем суммарную длину всех градиентов
grad_norm = sqrt( сумма(grad[i]²) по всем параметрам )

# Шаг 2: если слишком большая — масштабируем
if grad_norm > max_norm:
    for param in model.parameters():
        param.grad *= max_norm / grad_norm
```

Пример:

```
Обычный шаг: grad_norm = 0.8  → не трогаем (< 1.0)
Взрыв:       grad_norm = 7.3  → все градиенты уменьшаем в 7.3 раза
                               → grad_norm становится 1.0
```

Смысл: направление движения сохраняется (мы по-прежнему идём туда же), но размер шага ограничивается.

В логах нормальное обучение: `grad_norm ≈ 0.3 — 2.0`.
Если видите `grad_norm = 50.0` каждый шаг — что-то серьёзно не так.

### Экономия памяти: накопление градиентов

Идеальный батч для обучения языковых моделей — 128 примеров за раз.

Проблема: 128 примеров × 512 токенов × 4096 размерность × 3 мысли = много гигабайт памяти.

```
Batch=128 → требует ~18 ГБ только для активаций → OutOfMemory на 24 ГБ GPU!
```

**Gradient Accumulation** (накопление градиентов) — обходной путь:

```python
optimizer.zero_grad()  # обнулили накопленные градиенты

for micro_step in range(GRAD_ACCUM):  # 8 маленьких шагов
    x, y = get_batch()               # batch=16 (маленький)
    loss = model(x, y) / GRAD_ACCUM  # ← ВАЖНО: делим на 8!
    loss.backward()                  # градиенты накапливаются (+=)

# После 8 шагов накоплений:
optimizer.step()   # один большой шаг оптимизатора
optimizer.zero_grad()
```

Эффект: каждый форвард считается для batch=16 (помещается в память).
После 8 таких форвардов накопленные градиенты эквивалентны одному форварду с batch=128.

**Почему делим на GRAD_ACCUM?**

`.backward()` **добавляет** градиенты к уже накопленным (не заменяет).

Без деления на 8:
```
8 × gradient(batch=16) = 8 × маленький_градиент
```
Это в 8 раз больше чем должно быть! Взрыв градиента гарантирован.

С делением:
```
8 × (1/8 × gradient(batch=16)) = 1 × gradient(batch=128)
```
Математически эквивалентно одному большому батчу.

**Реальный баг из практики:**

Во время разработки v14.1 Олег однажды забыл разделить loss на GRAD_ACCUM.

```
Ожидаемый loss: ~4.5
Реальный loss:  ~4.5 × 8 = ~36.0  ← потому что накопили 8 раз без нормализации
```

Первые несколько шагов loss был странно огромным, но... шаги оптимизатора ведь были тоже большими. Система сама компенсировала? Не совсем.

Проблема была в **логировании**: в логах писали `total_loss` без умножения на GRAD_ACCUM:

```python
# Неправильно: loss уже делили на GRAD_ACCUM, но забыли умножить обратно в логах
bpb = total_loss / math.log(2)              # ← слишком маленькое число

# Правильно:
bpb = total_loss * GRAD_ACCUM / math.log(2) # ← умножаем чтобы получить "настоящий" loss
```

Итог: в логах BPB выглядел фантастически хорошим (2.1!) — но это была ошибка подсчёта.
Настоящий BPB был 4.2. После исправления логирования всё встало на место.

**Мораль:** всегда проверяй что числа в логах имеют смысл. BPB=2.1 для маленькой модели после 2000 шагов — слишком хорошо чтобы быть правдой.

### Быстрая математика: bfloat16

Всё обучение идёт в формате `bfloat16` (bf16).

Сравнение форматов:

```
float32:  ████████████████████████████████  32 бита
          ← 8 бит exp →← 23 бита мантисса →
          Диапазон: ±3.4 × 10³⁸  Точность: 7 значащих цифр

float16:  ████████████████  16 бит
          ← 5 бит exp →← 10 бит мантисса →
          Диапазон: ±65504  Точность: 3 значащих цифры
          Проблема: легко переполнение (NaN)!

bfloat16: ████████████████  16 бит
          ← 8 бит exp →← 7 бит мантисса →
          Диапазон: как float32!  Точность: 2 значащих цифры
          РЕШЕНО: диапазон большой, точность меньше — для нейросетей норм
```

Ключевая идея: для нейронных сетей **диапазон важнее точности**. Переполнение (NaN) убивает обучение мгновенно. Потеря пары значащих цифр — почти не заметна.

bfloat16 — лучшее из двух миров: скорость float16, диапазон float32.

```python
with torch.autocast(device_type='cuda', dtype=torch.bfloat16):
    logits, loss = model(x, y)
    # Внутри блока: матричные умножения в bf16 (быстро!)
    # PyTorch сам выбирает где нужна float32 (например, softmax)

# loss.backward() — PyTorch автоматически аккумулирует в float32 для точности
```

На современных GPU (A100, RTX 3090+) bf16 в 2× быстрее float32.

### Полный цикл обучения

Собираем всё вместе:

```python
step = 0
optimizer.zero_grad()

while step < TOTAL_STEPS:  # 40,000 шагов

    # === Шаг 1: вычислить LR для этого момента ===
    current_lr = cosine_lr(step, TOTAL_STEPS, MAX_LR, MIN_LR, WARMUP_STEPS)

    # Применяем к группам: кодбук в 10 раз меньше
    for i, group in enumerate(optimizer.param_groups):
        group['lr'] = current_lr if i == 0 else current_lr * 0.1

    # === Шаг 2: 8 маленьких батчей → накапливаем градиенты ===
    total_loss = 0.0
    for micro_step in range(GRAD_ACCUM):  # 8 раз
        x, y = get_batch('train')          # batch=16 каждый раз
        with torch.autocast('cuda', dtype=torch.bfloat16):
            logits, loss = model(x, y)
        loss = loss / GRAD_ACCUM           # нормализуем!
        loss.backward()                    # += градиент к накопленному
        total_loss += loss.item()

    # === Шаг 3: обрезаем взрывные градиенты ===
    grad_norm = torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)

    # === Шаг 4: один шаг оптимизатора ===
    optimizer.step()
    optimizer.zero_grad()

    # === Шаг 5: логируем ===
    if step % 50 == 0:
        # total_loss уже делился на GRAD_ACCUM, нужно умножить обратно
        real_loss = total_loss * GRAD_ACCUM
        bpb = real_loss / math.log(2)
        print(f"Шаг {step:5d}: loss={real_loss:.3f} bpb={bpb:.2f} "
              f"lr={current_lr:.2e} grad_norm={grad_norm:.2f}")

    step += 1
```

Один "шаг" = 8 форвардов (batch=16 каждый) + 8 backward + 1 шаг оптимизатора.
Эффективный батч: 16 × 8 = 128 примеров на обновление весов.

```

Шаг    50: loss=8.441  bpb=12.18 lr=1.50e-04 grad_norm=0.89  ← разгоняемся
Шаг   100: loss=7.312  bpb=10.55 lr=3.00e-04 grad_norm=0.71  ← вышли на полный LR
Шаг   500: loss=5.891  bpb=8.50  lr=2.89e-04 grad_norm=0.65  ← быстрое падение
Шаг  2000: loss=4.520  bpb=6.52  lr=2.52e-04 grad_norm=0.54  ← замедляется
Шаг 10000: loss=3.104  bpb=4.48  lr=1.72e-04 grad_norm=0.48  ← идёт
Шаг 40000: loss=2.917  bpb=4.21  lr=1.00e-05 grad_norm=0.41  ← финал
```

На что обращать внимание:
- `bpb` должен монотонно уменьшаться (иногда скачет, но тренд вниз)
- `grad_norm` в районе 0.3—2.0 — норма
- `lr` плавно уменьшается по косинусу
- Если `bpb` стоит на месте 1000+ шагов — что-то не так (см. главу 18)

### Итоги главы

Обучение — это не просто "уменьшать loss":

```
AdamW:
  - Adam: для каждого параметра свой адаптивный шаг (через m и v)
  - W: weight decay применяется отдельно от адаптации (честный штраф)
  - beta1=0.9, beta2=0.95: насколько помнить историю

Два режима параметров:
  - Основные:  lr=3e-4, weight_decay=0.1  (обычный режим)
  - Кодбук:    lr=3e-5, weight_decay=0.0  (осторожно: STE + биполярные)

Cosine schedule:
  - Warmup 100 шагов: разгоняемся 0 → max_lr
  - Cosine 40K шагов: плавно max_lr → min_lr (в 30 раз меньше)

Безопасность:
  - Gradient clipping: max_norm=1.0 — страховочный трос
  - Gradient accumulation: 8 × batch=16 = эффективный batch=128

Математика:
  - bf16: скорость + диапазон, без NaN-взрывов
  - loss / GRAD_ACCUM при накоплении — не забывать!
  - При логировании: bpb = loss * GRAD_ACCUM / log(2)
```
