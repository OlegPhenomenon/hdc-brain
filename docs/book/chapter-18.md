# Глава 18. Loss и метрики — как читать тренировочные логи

---

Максим — тренер по боксу.

У него есть боксёр которого он готовит к чемпионату. Как понять что боксёр становится лучше?

Не только смотреть на победы и поражения (это rare events). Нужны постоянные метрики:
скорость удара, точность, выносливость, время реакции.

Языковая модель — то же самое. Нельзя просто "поговорить" с ней каждые 1000 шагов.
Нужен числовой показатель который обновляется на каждом шаге.

Этот показатель — **loss**.

---

### Проблема: как измерить "понимание языка"

В задаче классификации всё просто: правильный ответ — 1, неправильный — 0. Считаем процент.

Но языковая модель предсказывает **следующий токен**. Правильных вариантов много:

```
"Я пошёл в ..." — что дальше? Магазин? Школу? Кино? Парк?
```

Нет одного правильного ответа. Нужен другой подход.

Идея: нейросеть должна давать **высокую вероятность** тому что реально встречается в тексте.

```
Тест на тексте "Столица Франции — Париж":

Плохая модель:   P("Париж") = 0.001  ← считает что это маловероятно
Средняя модель:  P("Париж") = 0.05   ← знает, но не уверена
Хорошая модель:  P("Париж") = 0.70   ← уверенно предсказывает
```

Чем выше вероятность реального следующего токена — тем лучше модель.

---

### Cross-Entropy Loss: формула

Нам нужно превратить "вероятность реального токена" в один скаляр.

Используем **отрицательный логарифм**:

```
loss = -log(P(реальный_токен))
```

Смотрим на числа:

```
P = 1.0  →  loss = -log(1.0) = 0.0      ← идеально
P = 0.5  →  loss = -log(0.5) = 0.69     ← неплохо
P = 0.1  →  loss = -log(0.1) = 2.30     ← плохо
P = 0.01 →  loss = -log(0.01) = 4.61    ← очень плохо
P = 0.001 → loss = -log(0.001) = 6.91   ← ужасно
```

Почему именно логарифм? Потому что он "штрафует" неуверенность нелинейно.
Снизить вероятность с 1.0 до 0.5 — небольшой штраф (0.69).
Снизить с 0.01 до 0.001 — большой штраф (от 4.61 до 6.91).

---

### В коде PyTorch

```python
import torch.nn.functional as F

# logits: предсказания модели (до softmax)
# targets: реальные следующие токены (индексы)

loss = F.cross_entropy(
    logits.view(-1, vocab_size),  # (B×T, V) — объединяем батч и время
    targets.view(-1)              # (B×T,)
)

# cross_entropy делает внутри:
# 1. softmax(logits) → вероятности
# 2. берёт P для реального токена
# 3. вычисляет -log(P)
# 4. усредняет по всем позициям
```

### BPB: удобная интерпретация

Cross-entropy loss — в натуральных логарифмах (log base e). Это математически удобно.

Но что значит `loss = 3.5`? Плохо это или хорошо?

**Переводим в биты:** из информационной теории известно что оптимальный код
для события с вероятностью P требует `-log₂(P)` бит.

```
BPB = loss / log(2)
```

Перевод из "натов" (единица в натуральных логарифмах) в биты.

"Bits Per Byte" — сколько бит нужно чтобы закодировать один байт текста
если использовать нашу модель как компрессор.

```python
import math

loss = 3.5
bpb = loss / math.log(2)  # = 3.5 / 0.693 = 5.05
print(f"BPB = {bpb:.2f}")
```

---

### Шкала BPB: что хорошо, что плохо

```
BPB = 1.0    теоретический минимум (недостижимо — язык случаен)
BPB = 2.0    очень хорошо (уровень GPT-4 для английского)
BPB = 3.0    хорошо (GPT-2 Large: 32 слоя, 774M параметров)
BPB = 4.0    приемлемо (GPT-2 Small: 117M параметров)
BPB = 5.0    слабо — понимает структуру, теряет смысл
BPB = 6.0    очень слабо — едва понимает язык
BPB = 7.0+   почти случайное угадывание
```

Для HDC-Brain v14.1:
```
Начало обучения:  BPB ≈ 7.5   (почти случайный)
Середина (20K):   BPB ≈ 5.5
Конец (40K):      BPB ≈ 4.2   (приемлемо!)
Цель (100K+):     BPB ≈ 3.5   (хорошо)
```

### Validation Loss: проверка на честность

Есть опасная ловушка: модель может **заучить** тренировочные данные.

```
Тренировочный текст: "The capital of France is Paris."

Если модель просто запомнила:
  P("Paris" | "The capital of France is") = 0.99

Train BPB = 2.0  ← выглядит отлично!
Но на новом тексте:  BPB = 7.0  ← провал
```

Это называется **переобучение (overfitting)**.

Защита: держим **валидационный набор** который модель никогда не видела:

```
Все данные:   90% → тренировка,  10% → валидация

Каждые 500 шагов:
  Train BPB:   4.1   (на тренировочных данных)
  Val BPB:     4.3   (на невиданных данных)
  Разница 0.2 — нормально!

Признак переобучения:
  Train BPB:   2.5
  Val BPB:     6.8   ← огромная разница — проблема!
```

В нашей модели переобучение маловероятно: данные огромные (гигабайты),
каждый пример видим один раз.

### Tokens Per Second и ETA

Помимо качества важна **скорость**. В логах:

```
Step  100 | loss=6.834 | bpb=9.86  | tok/s=38,124 | lr=6e-05
```

**tok/s (Tokens Per Second)** — скорость обработки:

```python
tokens_per_step = batch_size × seq_len × grad_accum
# = 16 × 512 × 8 = 65,536 токенов за один шаг

elapsed = time.time() - step_start
tps = tokens_per_step / elapsed
```

С нашими настройками: ~38,000 токенов/сек на RTX 3090.

**ETA (Estimated Time Arrival)** — сколько ещё ждать:

```python
steps_remaining = total_steps - current_step
time_per_step = elapsed / current_step
eta = steps_remaining × time_per_step
```

При 38K tok/s и 40,000 шагах: ~24 часа обучения.

### Реальный кейс: когда модель не учится

При первом запуске v14.1 мы наблюдали странное:

```
Шаг    0: loss ≈ 10.4   (случайный)
Шаг  500: loss = 7.46   (упал немного)
Шаг 1000: loss = 7.47   (СТОП! Не двигается)
Шаг 2000: loss = 7.47   (два часа GPU — впустую)
Шаг 2800: loss = 7.47   (всё ещё плато)
```

Генерация:
```
"The capital of France is2 as the you1 the and. a a or to0"
```

Полный мусор. Модель выучила только что "the", "a", "of" — самые частые токены.

**Диагностика:**

Смотрим на масштаб логитов:
```python
print(f"Logits std: {logits.std().item():.4f}")
# → Logits std: 0.0198   ← ОЧЕНЬ маленький!
```

Проблема была в одной строке инициализации:
```python
self.output_scale = nn.Parameter(torch.tensor(hdc_dim ** -0.5))
# hdc_dim = 4096 → output_scale = 1/√4096 = 0.0156
```

Логиты умножались на 0.0156 → стандартное отклонение ≈ 0.02.

При таких маленьких логитах softmax даёт почти равномерное распределение:
все 32,000 токенов имеют вероятность ≈ 1/32,000.
Модель не может "различить" токены!

**Фикс:**
```python
# Было:
self.output_scale = nn.Parameter(torch.tensor(hdc_dim ** -0.5))  # → 0.0156

# Стало:
self.output_scale = nn.Parameter(torch.tensor(1.0))  # нормальный масштаб
```

**После фикса:**
```
Шаг  500: loss = 7.33   ← уже лучше чем старое плато!
Шаг 1000: loss = 6.55   ← пробили барьер 7.0
Шаг 2000: loss = 5.87
Шаг 6000: loss = 5.08
```

Генерация на шаге 6000:
```
"The capital of France is a great way to do this. That's why
we're not all that our ancestors have been the last..."
```

Грамматика! Апострофы! Связный текст!

**Урок:** одно число `0.0156` вместо `1.0` убивает обучение всей модели.

Как диагностировать:
```python
# После каждых 50 шагов в первые 500:
print(f"Logits std:   {logits.std().item():.3f}")      # должно быть 1-10
print(f"Max softmax:  {F.softmax(logits,-1).max():.3f}") # должно быть > 0.01
```

### Что делать если что-то пошло не так

```
Loss застрял с самого начала (~8.0):
  → learning rate слишком большой, градиент взрывается
  → Признак: loss прыгает вверх-вниз
  → Решение: уменьши LR в 5-10×, проверь gradient clipping

Loss быстро падает, потом плато:
  → LR стал слишком маленьким (schedule умер)
  → Признак: оба train и val не меняются
  → Решение: проверь cosine schedule

Loss падает, val растёт:
  → Переобучение
  → Решение: больше данных, добавь dropout

Loss = NaN:
  → overflow в bf16 или деление на 0
  → Признак: с первых шагов
  → Решение: clip gradients агрессивнее, уменьши LR

Tokens/s неожиданно низкий (< 20K):
  → DataLoader узкое место или checkpointing без CUDA
  → Решение: увеличь num_workers, проверь device
```

### Формула loss (полная)

```
CrossEntropy(logits, targets) =
  -1/N × Σ log(softmax(logits[i])[targets[i]])

где:
  N — количество позиций (B × T)
  logits[i] — вектор предсказаний для позиции i
  targets[i] — реальный следующий токен на позиции i
  softmax(·)[j] — вероятность j-го токена
```

Вот что каждый символ означает:
- `softmax(logits[i])` — превращаем "сырые" баллы в вероятности (сумма = 1)
- `[targets[i]]` — берём вероятность именно правильного токена
- `log(...)` — логарифм этой вероятности (отрицательный = штраф)
- `Σ / N` — усредняем по всем позициям

### Итоги главы

Как измеряем качество:

```
Cross-entropy loss:  -log(P реального токена)
                     0 = идеально, растёт с ошибками

BPB = loss / log(2): биты на байт
                     < 4 = хорошо, > 6 = плохо

Validation loss:     честная проверка без переобучения
Tokens/s:            скорость обучения
ETA:                 сколько ещё ждать
```

Красные флаги:
- Logits std < 0.1 (слишком маленький output_scale)
- Loss = NaN (overflow, gradient explode)
- Train BPB << Val BPB (переобучение)
- Tokens/s < 20K (узкое место в данных или checkpointing)
