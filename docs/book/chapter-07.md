## История: Денис и фотографии в пикселях

Денис — фотограф. Он хранит RAW-фотографии — это TIFF файлы, 48 МБ каждый. У него 10 000 фотографий. Это 480 ГБ.

Коллега говорит: "Денис, тебе правда нужна такая точность? Ты фотографируешь пейзажи, а не научные микрографии."

Денис думает: "Моему глазу хватит 8 бит на канал вместо 16. Сожму вдвое."

Результат: 240 ГБ. Глаз не видит разницы.

Потом кто-то предлагает JPEG с лёгким сжатием — 24 МБ вместо 48 МБ. Снова Денис не видит разницы.

Это **квантизация** — уменьшение точности данных с минимальной потерей качества.

В нейросетях то же самое. Числа хранятся как float32 (32 бита). Но нужны ли все 32 бита?

---

## 32 бита — не всегда нужны

Посмотри на одно число: `0.37182649`

Оно занимает 32 бита в памяти:

```
0.37182649 в float32:
  ┌─┬────────┬───────────────────────┐
  │0│01111101│01111110001000001100101│
  └─┴────────┴───────────────────────┘
  ↑     ↑              ↑
 знак экспонента    мантисса
 (1 бит) (8 бит)   (23 бита)
```

Для HDC нам интересна только одна вещь: **на каком полюсе это число?** Положительное или отрицательное?

```
0.37182649  → ПОЛОЖИТЕЛЬНОЕ → +1
-0.82614    → ОТРИЦАТЕЛЬНОЕ → -1
0.00001     → ПОЛОЖИТЕЛЬНОЕ → +1
-100000.0   → ОТРИЦАТЕЛЬНОЕ → -1
```

Это называется **bipolar** (двуполярное) — всего два значения: +1 и -1.

Нам нужен **1 бит** вместо 32.

---

## Что теряем и что получаем

```
ТЕРЯЕМ:
─────────────────────────────────────
  Точность: 0.9 и 0.1 → оба +1 (неразличимы!)
  Разнообразие: только 2 значения вместо 4 миллиардов


ПОЛУЧАЕМ:
─────────────────────────────────────
  Память: в 32 раза меньше!
  Скорость: операции над битами — самые быстрые на любом CPU/GPU
  Устойчивость: нет overflow, нет NaN, нет числовых проблем
```

Давай посчитаем разницу для нашего Codebook:

```
Codebook: 32,000 токенов × 4,096 чисел

Float32:  32,000 × 4,096 × 4 байта  =  524,288,000 байт  ≈  500 МБ
Bipolar:  32,000 × 4,096 × 1 бит    =   16,384,000 байт  ≈   16 МБ
                                                           ↑
                                              в 32 раза меньше!
```

16 МБ вместо 500 МБ. Это важно: 16 МБ **влезает в кэш CPU**. Кэш = мгновенный доступ без задержек.

## Почему умножение ±1 такое быстрое

На современных процессорах есть специальные операции над битами. Самая полезная для нас — **XNOR** (исключающее НЕ-ИЛИ).

Посмотри на таблицу умножения для ±1:

```
(+1) × (+1) = +1   ← оба одинаковые → +1
(+1) × (-1) = -1   ← разные → -1
(-1) × (+1) = -1   ← разные → -1
(-1) × (-1) = +1   ← оба одинаковые → +1
```

Это в точности совпадает с операцией XNOR (заменяем +1→1, -1→0):

```
1 XNOR 1 = 1   ← совпали → 1
1 XNOR 0 = 0   ← не совпали → 0
0 XNOR 1 = 0   ← не совпали → 0
0 XNOR 0 = 1   ← совпали → 1
```

Одна инструкция XNOR на процессоре обрабатывает **64 пары бит за раз** (64-битный регистр). Обычное float-умножение — одна пара за раз.

Ускорение скалярного произведения:

```
Обычный dot product (float32): ~1×   (базовая скорость)
XNOR + POPCNT (бинарный):      ~64×  (на CPU)
                                ~100-1000× (на специальных чипах)
```

> **POPCNT** — "population count", считает количество единиц в числе.
> Нам нужно посчитать сколько битов совпало (XNOR), потом POPCNT считает совпадения.
> Итог: скалярное произведение за наносекунды.

## Почему ±1, а не {0, 1}

Можно было использовать двоичный код: 0 и 1. Чем bipolar лучше?

```
Обычный бинарный {0, 1}:
  Среднее случайного вектора = 0.5   (смещено!)
  Скалярное произведение двух случайных векторов > 0

Bipolar {-1, +1}:
  Среднее случайного вектора = 0.0   (нет смещения!)
  Скалярное произведение двух случайных векторов ≈ 0
```

Bipolar симметричен относительно нуля. Это означает:
- Случайные векторы автоматически ортогональны (сходство ≈ 0)
- Bind не вносит систематического смещения
- Математика чище

```python
import torch

dim = 4096
n = 1000

# Сравним два подхода
binary  = torch.randint(0, 2, (n, dim)).float()     # {0, 1}
bipolar = torch.sign(torch.randn(n, dim))            # {-1, +1}

# Среднее попарных скалярных произведений:
def avg_dot(vectors):
    # Берём первые 100 пар
    total = 0
    for i in range(100):
        total += torch.dot(vectors[i], vectors[i+1]).item()
    return total / 100 / dim

print(f"Binary  avg dot: {avg_dot(binary):.4f}")    # ≈ 0.25 (смещено!)
print(f"Bipolar avg dot: {avg_dot(bipolar):.4f}")   # ≈ 0.0  (нет смещения)
```

## Красивый факт: Hamming = Cosine

Для bipolar векторов есть красивое математическое соотношение.

**Hamming distance** — количество позиций где векторы отличаются:

```
A = [+1, -1, +1, -1, +1]
B = [+1, +1, +1, -1, -1]
         ↑              ↑
       отличается    отличается → Hamming(A, B) = 2
```

Оказывается, для bipolar векторов:

```
Cosine_similarity(A, B)  =  1 - 2 × Hamming(A, B) / D
```

Или наоборот:

```
Hamming(A, B)  =  D × (1 - Cosine(A, B)) / 2
```

Таблица соответствия:

```
Hamming    Cosine   Интерпретация
─────────────────────────────────
0 (0%)     +1.0     Одинаковые
D/4 (25%)  +0.5     Похожие
D/2 (50%)   0.0     Ортогональные (случайные)
3D/4 (75%) -0.5     Непохожие
D (100%)   -1.0     Противоположные
```

**Зачем это важно?** Cosine similarity (геометрическая мера) = функция от Hamming (битовой операции). Можно всё считать через биты!

## Проблема: как обучать бинарные веса?

Вот главная трудность.

Мы хотим веса {-1, +1}. Но нейросеть учится через **градиенты** — производные функции потерь по параметрам.

Посмотри на функцию `sign(x)`:

```
График sign(x):

  +1 ──────────────────────╮
                           │ ← скачок в нуле
  -1 ╭─────────────────────
     |
     0
```

Производная? Везде ноль, в точке 0 — бесконечность:

```
d/dx sign(x) = 0  при  x ≠ 0   ← нулевая производная везде!
               ∞  при  x = 0
```

Нулевая производная = нет сигнала для обучения. Нейросеть не может учиться.

```
Параметр w = 0.73
sign(0.73) = +1

Градиент через sign: d(loss)/d(0.73) = 0   ← стоп! ничего не передаётся
```

Это как сказать Олегу: "Я не скажу тебе было ли лучше или хуже. Удачи с уткой."

Решение этой проблемы — в следующей главе (STE). Сначала запомним: bipolar быстро и компактно, но требует специального трюка для обучения.

## Alpha-scaling: немного умнее чем просто знак

В нашем коде мы не просто берём `sign(w)`. Мы делаем `alpha * sign(w)`:

```python
alpha = torch.mean(torch.abs(w), dim=-1, keepdim=True)
hard  = alpha * torch.sign(w)
```

Зачем `alpha`?

Посмотри на два вектора:

```
вектор A: [0.001, -0.002, 0.001, -0.001]
вектор B: [100,   -200,   100,   -100  ]
```

После `sign()` — они одинаковые: `[+1, -1, +1, -1]`.

Но их **масштабы** совершенно разные! A — тихий шёпот, B — крик.

`alpha` = среднее абсолютное значение = "типичная громкость" вектора:

```
alpha_A = mean(|0.001|, |0.002|, |0.001|, |0.001|) = 0.00125
alpha_B = mean(|100|, |200|, |100|, |100|) = 125

alpha_A × sign(A) = [0.00125, -0.00125, 0.00125, -0.00125]
alpha_B × sign(B) = [125, -125, 125, -125]
```

Теперь масштаб сохранился! Это называется **alpha-scaling**.

```python
def alpha_sign(w):
    """Бинаризация с сохранением масштаба."""
    alpha = w.abs().mean(dim=-1, keepdim=True)
    return alpha * w.sign()


a = torch.tensor([0.001, -0.002, 0.001, -0.001])
b = torch.tensor([100.0, -200.0, 100.0, -100.0])

print("Просто sign:")
print(a.sign().tolist())   # [1, -1, 1, -1]
print(b.sign().tolist())   # [1, -1, 1, -1]  ← одинаково!

print("\nAlpha-scaling:")
print(alpha_sign(a).tolist())   # [0.00125, -0.00125, ...]
print(alpha_sign(b).tolist())   # [125, -125, ...]  ← разные!
```

## Будущее: модель в 37 МБ на телефоне

Сейчас мы тренируем в float32/bf16, при inference используем bipolar. Но можно пойти дальше.

**1-bit inference**: сохраняем веса не как float (4 байта), а как биты (1/8 байта):

```
Обычная модель 300M:  300M × 4 байта = 1.2 GB
1-bit модель 300M:    300M × 1 бит   = 37.5 MB  ← в 32 раза меньше!
```

37 МБ — помещается в оперативную память смартфона. Модель работает офлайн без интернета.

Именно это делает **BitNet** (2023, Microsoft) — они обучили трансформер в 1 бит и он конкурирует с float16 моделью вдвое большего размера.

Наша архитектура идёт ещё дальше: бинарность не только в весах но и в самих операциях (Bind = XNOR).

### Задание 7.1 — Три способа бинаризации

Сравни три способа перейти от float к bipolar:

```python
import torch

dim = 4096
original = torch.randn(dim)   # float32

# Способ 1: просто sign
binary_1 = torch.sign(original)

# Способ 2: alpha-scaling
alpha = original.abs().mean()
binary_2 = alpha * torch.sign(original)

# Способ 3: среднее через медиану вместо среднего
median = original.abs().median()
binary_3 = median * torch.sign(original)

# Для каждого: косинусное сходство с original
def cosine(a, b):
    return (torch.dot(a, b) / (a.norm() * b.norm())).item()

print(f"plain sign:    cos={cosine(original, binary_1):.4f}")
print(f"alpha-scaling: cos={cosine(original, binary_2):.4f}")
print(f"median-scaling: cos={cosine(original, binary_3):.4f}")

```

### Задание 7.2 — Проверь формулу Hamming = Cosine

```python
import torch

def hamming(a, b):
    """Количество позиций где векторы отличаются."""
    return (a != b).sum().item()

def cosine_sim(a, b):
    return (torch.dot(a.float(), b.float()) /
            (a.float().norm() * b.float().norm())).item()

dim = 4096
errors = []

for _ in range(100):
    a = torch.sign(torch.randn(dim))
    b = torch.sign(torch.randn(dim))

    H = hamming(a, b)
    cos_from_formula = 1 - 2 * H / dim   # наша формула
    cos_real = cosine_sim(a, b)

    errors.append(abs(cos_from_formula - cos_real))

print(f"Средняя ошибка формулы: {sum(errors)/len(errors):.8f}")
# Должна быть почти нулевой!
```

import torch, time
