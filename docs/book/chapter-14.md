# Глава 14. HDC Binding Attention — наш главный трюк

---

В главе 13 мы увидели проблему:

Стандартный MultiHead Attention при D=4096 требует **67 миллионов параметров на один слой**.
Если у нас 8 слоёв — это 536M параметров только на attention.
А вся наша модель — 299M.

Что-то не сходится.

Значит, стандартный attention нам не подходит. Нужен другой способ.

---

### Что делает матрица W_q на самом деле?

В стандартном attention мы вычисляем запросы так:

```python
Q = x @ W_q   # (T, D) @ (D, D) = (T, D)
```

Что делает матрица `W_q`?

Представим её как машину для смешивания:

```
Q[i] = W_q[i, 0] × x[0]
      + W_q[i, 1] × x[1]
      + W_q[i, 2] × x[2]
      + ...
      + W_q[i, D-1] × x[D-1]
```

Каждое измерение выхода Q — **взвешенная комбинация всех измерений x**.
Любое из 4096 измерений x влияет на любое из 4096 измерений Q.

Это называется "cross-dimension mixing" — перемешивание измерений.

Сильная операция. Очень много параметров.

---

### Ключевой вопрос: а обязательно нужно перемешивать?

Вспомним операцию **Bind** из главы 6:

```python
Q = x * bv_q   # поэлементное умножение
```

Каждое измерение выхода зависит **только от того же измерения x**:

```
Q[0] = x[0] × bv_q[0]
Q[1] = x[1] × bv_q[1]
Q[i] = x[i] × bv_q[i]
```

Никакого перемешивания. Каждое измерение — сам по себе.

Зато `bv_q` — это **вектор**, а не матрица:
- Матрица W_q: D × D = 4096 × 4096 = **16.7M параметров**
- Вектор bv_q: D = 4096 = **4096 параметров**

Разница: в 4096 раз!

Вот в чём главная идея HDC Binding Attention:
**заменяем матричные проекции поэлементным умножением на обученный вектор**.

---

### Интуиция: что делает Bind

Bind — это "покраска" вектора.

Представь что исходный токен — белый лист бумаги, на котором написан текст.

- `bv_q` — "красный фильтр" (для запросов)
- `bv_k` — "синий фильтр" (для ключей)
- `bv_v` — "зелёный фильтр" (для значений)

Когда мы делаем `Q = x * bv_q` — мы "смотрим на токен через красный фильтр".
Когда делаем `K = x * bv_k` — "через синий".

Через разные фильтры один и тот же текст выглядит по-разному.
Это и даёт разные "точки зрения" для Q, K, V — без матрицы!

---

### Binding vectors — бинарные через STE

Наши `bv_q`, `bv_k`, `bv_v` — не просто векторы. Они **бинарные** (через STE из главы 8):

```python
class MultiHeadBindingAttention(nn.Module):
    def __init__(self, hdc_dim, n_heads=4):
        super().__init__()
        self.hdc_dim = hdc_dim        # 4096
        self.n_heads = n_heads        # 4
        self.head_dim = hdc_dim // n_heads  # 1024

        # Параметры: только 3 вектора на H голов (не матрицы!)
        self.bv_q = nn.Parameter(torch.randn(n_heads, self.head_dim) * 0.02)  # (4, 1024)
        self.bv_k = nn.Parameter(torch.randn(n_heads, self.head_dim) * 0.02)  # (4, 1024)
        self.bv_v = nn.Parameter(torch.randn(n_heads, self.head_dim) * 0.02)  # (4, 1024)
        # Итого: 3 × 4 × 1024 = 12,288 параметров
```

Сравним с обычным MHA:

```
Обычный MHA:       W_q + W_k + W_v + W_o = 4 × 4096² = 67,108,864 параметров
HDC Binding:       bv_q + bv_k + bv_v    = 3 × 4096  = 12,288 параметров

Экономия: в 5461 раз!
```

### Многоголовый вариант: разбиваем вектор на сектора

Как сделать несколько "голов" без матрицы?

Разбиваем D-мерный вектор токена на H равных кусков:

```
Вектор токена [d₀, d₁, ..., d₄₀₉₅]

Голова 1: [d₀,    d₁,    ..., d₁₀₂₃]   ← первые 1024 измерений
Голова 2: [d₁₀₂₄, d₁₀₂₅, ..., d₂₀₄₇]  ← следующие 1024
Голова 3: [d₂₀₄₈, d₂₀₄₉, ..., d₃₀₇₁]
Голова 4: [d₃₀₇₂, d₃₀₇₃, ..., d₄₀₉₅]
```

Каждая голова работает в своём "секторе" вектора.
У каждой головы свой bv_q, bv_k, bv_v:

```
Голова 1: bv_q₁ (1024 чисел), bv_k₁, bv_v₁
Голова 2: bv_q₂ (1024 чисел), bv_k₂, bv_v₂
Голова 3: bv_q₃, bv_k₃, bv_v₃
Голова 4: bv_q₄, bv_k₄, bv_v₄
```

Каждая голова смотрит на текст через свой фильтр, в своём секторе.

---

### Полный код MultiHeadBindingAttention

```python
class MultiHeadBindingAttention(nn.Module):
    def __init__(self, hdc_dim, n_heads=4):
        super().__init__()
        self.n_heads = n_heads
        self.head_dim = hdc_dim // n_heads  # 1024
        self.scale = self.head_dim ** -0.5  # 1/√1024 = 1/32 (нормализация)

        self.bv_q = nn.Parameter(torch.randn(n_heads, self.head_dim) * 0.02)
        self.bv_k = nn.Parameter(torch.randn(n_heads, self.head_dim) * 0.02)
        self.bv_v = nn.Parameter(torch.randn(n_heads, self.head_dim) * 0.02)

    def _ste_sign(self, w):
        """Бинаризуем вектор: хранится float, используется ±alpha."""
        alpha = torch.mean(torch.abs(w), dim=-1, keepdim=True)
        hard = alpha * torch.sign(w)
        hard = torch.where(hard == 0, alpha * torch.ones_like(hard), hard)
        return (hard - w).detach() + w

    def forward(self, x):
        B, T, D = x.shape   # например (12, 512, 4096)
        H = self.n_heads     # 4
        HD = self.head_dim   # 1024

        # === Шаг 1: Бинаризуем binding vectors (STE) ===
        bv_q = self._ste_sign(self.bv_q)  # (4, 1024) — значения ±alpha
        bv_k = self._ste_sign(self.bv_k)
        bv_v = self._ste_sign(self.bv_v)

        # === Шаг 2: Разбиваем токены на головы ===
        x_heads = x.view(B, T, H, HD)
        # (12, 512, 4, 1024) — каждый токен разделён на 4 куска

        # === Шаг 3: BIND — поэлементное умножение ===
        # bv_q: (4, 1024) → broadcast по B и T → (12, 512, 4, 1024)
        Q = x_heads * bv_q  # каждая голова смотрит через свой фильтр
        K = x_heads * bv_k
        V = x_heads * bv_v

        # === Шаг 4: Перестановка осей для attention ===
        Q = Q.permute(0, 2, 1, 3)  # (B, H, T, HD) = (12, 4, 512, 1024)
        K = K.permute(0, 2, 1, 3)
        V = V.permute(0, 2, 1, 3)

        # === Шаг 5: Scores ===
        scores = Q @ K.transpose(-2, -1) * self.scale  # (12, 4, 512, 512)

        # === Шаг 6: Causal mask ===
        causal = torch.tril(torch.ones(T, T, device=x.device, dtype=torch.bool))
        scores = scores.masked_fill(~causal, float('-inf'))

        # === Шаг 7: SIGMOID вместо Softmax! ===
        attn = torch.sigmoid(scores * 4.0)   # (12, 4, 512, 512)
        attn = attn.masked_fill(~causal, 0.0)  # явно зануляем будущее

        # === Шаг 8: Взвешенная сумма V ===
        out = attn @ V   # (12, 4, 512, 1024)

        # === Шаг 9: Объединяем головы ===
        out = out.permute(0, 2, 1, 3).contiguous().view(B, T, D)
        # (12, 4, 512, 1024) → (12, 512, 4, 1024) → (12, 512, 4096)
        # Каждая голова писала в свой сектор → просто собираем обратно

        return out
```

### Почему нет out_projection (W_o)?

В стандартном MHA после объединения голов идёт финальная матрица:

```python
# Стандартный:
out = concat_heads(...)  # (B, T, D)
out = self.W_o(out)      # W_o: 4096 × 4096 = 16.7M параметров!
```

`W_o` нужен чтобы "перемешать" информацию от разных голов — голова 1 хранит одно,
голова 2 другое, и нужно их скомбинировать.

Но в нашей архитектуре каждая голова работает в **своём секторе** D-вектора:
- Голова 1 → измерения 0..1023
- Голова 2 → измерения 1024..2047
- Голова 3 → измерения 2048..3071
- Голова 4 → измерения 3072..4095

Когда мы объединяем головы через `view(B, T, D)` — это просто "склеивание" секторов обратно.
Каждая голова уже пишет в свою "зону" — перемешивать не нужно.

Экономия: ещё 16.7M параметров.

---

### Sigmoid вместо Softmax: в чём разница

Это важный момент, который отличает нас от стандартного attention.

**Softmax** — конкурентный:
```python
softmax([2.0, 1.0, 0.5]) = [0.66, 0.24, 0.10]
#                            сумма всегда = 1.0
```
Один токен "выигрывает" — остальные получают меньше.
Если один стал важнее, другие автоматически стали менее важными.

**Sigmoid** — независимый:
```python
sigmoid([2.0, 1.0, 0.5]) = [0.88, 0.73, 0.62]
#                             сумма = 2.23, не фиксирована
```
Каждый токен отвечает независимо: "достаточно ли я важен?"
Несколько токенов могут быть одновременно важными.

Зачем нам sigmoid?

В HDC идеология такая: **накапливаем сигналы**, а не выбираем победителя.
Это больше соответствует Bundle операции из главы 6.

Плюс: `* 4.0` добавляет "решительности":
```
sigmoid(1.0)     = 0.73
sigmoid(1.0 × 4) = 0.98   ← чёткий ответ
sigmoid(-1.0 × 4) = 0.02  ← чёткий отказ
```

### Считаем параметры

```
Стандартный Multi-Head Attention (D=4096, H=4):
  W_q: 4096 × 4096 = 16,777,216
  W_k: 4096 × 4096 = 16,777,216
  W_v: 4096 × 4096 = 16,777,216
  W_o: 4096 × 4096 = 16,777,216
  ─────────────────────────────────
  ИТОГО: 67,108,864 параметров ≈ 67M

HDC Binding Attention (D=4096, H=4):
  bv_q: 4 × 1024  = 4,096
  bv_k: 4 × 1024  = 4,096
  bv_v: 4 × 1024  = 4,096
  (нет W_o)
  ─────────────────────────────────
  ИТОГО: 12,288 параметров ≈ 12K

Отношение: 67,108,864 / 12,288 = 5,461×
```

В **5461 раз** меньше параметров в Attention.

---

### Честный анализ: что мы теряем

Чудес не бывает. За экономию мы платим.

**Нет cross-dimension mixing**

В стандартном attention измерение Q[0] зависит от всех измерений x.
У нас Q[0] зависит только от x[0].

Это значит: один проход через Binding Attention выразительнее слабее.
Компенсируем: 8 блоков × 3 прохода мысли = 24 прохода через данные.

**Binding vectors одинаковы для всех токенов**

`bv_q` — один вектор для всех токенов в словаре.
В стандартном attention W_q адаптируется к содержанию каждого токена.

У нас "фильтр" одинаков для всех — только сами токенные векторы разные.

**Меньше выразительности при той же размерности**

Стандартный attention D=4096 выразительнее нашего D=4096.
Но наш использует на 5000× меньше параметров — значит оставшиеся параметры
можно потратить на большее количество блоков, больший кодбук, больший контроллер.

---

### Формула HDC Binding Attention

```
Q = x * bv_q       ← Bind (поэлементное умножение)
K = x * bv_k
V = x * bv_v

scores = Q @ K^T × scale      scale = 1/√HD
attn   = sigmoid(scores × 4)  (с causal mask)
output = attn @ V
```

Вот что каждый символ означает буквально:
- `x * bv_q` — каждое измерение x умножается на соответствующее измерение bv_q
- `Q @ K^T` — матрица похожести: каждый запрос сравнивается с каждым ключом
- `× scale` — нормализация дисперсии (как в стандартном attention)
- `sigmoid(× 4)` — превращаем похожесть в вес [0..1], с * 4 для большего контраста
- `@ V` — взвешиваем значения по получившимся весам

### Итоги главы

HDC Binding Attention — замена стандартного Attention через HDC-операцию Bind:

```
Обычно:  Q = x @ W_q        — матрица 4096×4096 = 16.7M параметров
У нас:   Q = x * bv_q       — вектор 4096 = 4096 параметров

Итого: 12,288 параметров vs 67,108,864 = в 5461 раз меньше!
```

Ключевые отличия от стандартного MHA:
- Bind вместо матричной проекции
- Нет out_projection (каждая голова пишет в свой сектор)
- Sigmoid вместо Softmax (накапливаем, не конкурируем)

Компромисс: теряем cross-dimension mixing.
Компенсируем: 8 блоков × 3 мысли = 24 прохода через данные.
