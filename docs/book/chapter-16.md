# Глава 16. Thought Loops — многопроходное мышление

---

Представь что ты решаешь сложную задачу.

Первый раз читаешь условие — понимаешь общую картину.
Второй раз — замечаешь детали которые пропустил.
Третий раз — видишь связи между деталями.

Три прохода, одна и та же задача. Понимание углубляется.

Обычный трансформер так не умеет. Он читает один раз — и всё.

---

### Как трансформер "думает" обычно

В обычном трансформере есть N слоёв. Каждый слой — один проход:

```
Токены → Слой 1 → Слой 2 → ... → Слой N → Предсказание
```

Слой 1 обрабатывает токены.
Слой 2 обрабатывает результат Слоя 1.
...
И так далее. Одно направление. Вперёд и только вперёд.

**Проблема:** если что-то пошло не так в Слое 3 — исправить нельзя.
Слои 4-N работают с испорченным результатом.

---

### Universal Transformer: переиспользуем слои

В 2018 году Dehghani и коллеги придумали **Universal Transformer**.

Идея: вместо N разных слоёв — один слой, применённый N раз.

```
Без Universal Transformer:
  x → Слой1 → Слой2 → Слой3 → ...

С Universal Transformer:
  x → Слой → Слой → Слой → ...
  (один и тот же слой!)
```

Зачем? Слой "видит" свои предыдущие результаты и может их уточнить.
Это похоже на итеративное решение задачи.

Мы развили эту идею дальше: добавили механизм **learned gates** —
модель сама решает насколько "слушать" каждый дополнительный проход.

---

### Thought Loops: наша реализация

В HDC-Brain у нас **8 блоков** и **3 "мысли"** (прохода).

```
Мысль 1: токены → Block1 → Block2 → ... → Block8 → h1
Мысль 2: h1     → Block1 → Block2 → ... → Block8 → h2 → h = mix(h1, h2)
Мысль 3: h      → Block1 → Block2 → ... → Block8 → h3 → h = mix(h, h3)
```

Ключевое: те же самые 8 блоков используются на каждом проходе.
Это **не** 24 разных блока. Это 8 блоков, примененных 3 раза.

Параметры блоков — одни и те же. Просто прогоняем данные через них снова.

---

### Thought Position: "на какой мысли я?"

Проблема: если на каждом проходе подаём одинаковый вход — блоки не знают
"на первом проходе я" или "на третьем".

Решение: перед каждым дополнительным проходом добавляем "позицию мысли":

```python
# Обучаемые векторы — по одному на каждую мысль
self.thought_pos = nn.Parameter(torch.randn(max_thoughts, hdc_dim) * 0.01)

# Перед вторым проходом:
thought_input = LayerNorm(h) + thought_pos[1]
# Перед третьим:
thought_input = LayerNorm(h) + thought_pos[2]
```

Теперь блоки получают разный сигнал в зависимости от номера мысли.
Они могут "адаптироваться": на первом проходе — анализировать, на третьем — финализировать.

Параметров: `max_thoughts × hdc_dim = 4 × 4096 = 16,384` — совсем мало.

---

### Learned Gates: сколько "слушать" эту мысль?

Главная инновация: модель сама решает насколько использовать каждый дополнительный проход.

```python
# Обучаемые скалярные параметры (по одному числу на каждую мысль)
self.thought_gates = nn.Parameter(torch.zeros(max_thoughts))
# zeros → sigmoid(0) = 0.5 — начинаем нейтрально
```

При каждом дополнительном проходе:

```python
gate = torch.sigmoid(self.thought_gates[t])  # число от 0 до 1

# Гейтированная интерполяция:
h = h + gate × (thought - h)
  = (1 - gate) × h  +  gate × thought
```

Это линейная интерполяция (перемешивание) между старым и новым результатом:

```
gate = 0.0  →  h = h          (новая мысль бесполезна, игнорируем)
gate = 0.5  →  h = 0.5×h + 0.5×thought  (пополам)
gate = 1.0  →  h = thought    (новая мысль полностью заменяет)
```

**Как gate учится?**

Если дополнительный проход помогает предсказать следующий токен —
градиент увеличит gate.
Если мешает — уменьшит.

Начинаем с 0.5. После обучения gate находит оптимальный баланс автоматически.

### Полный код ThoughtLoop

```python
import torch
import torch.nn as nn
from torch.utils.checkpoint import checkpoint as torch_checkpoint

class ThoughtLoop(nn.Module):
    def __init__(self, hdc_dim, max_thoughts=4):
        super().__init__()
        self.max_thoughts = max_thoughts

        # Один scalar gate на каждую мысль
        # zeros → sigmoid(0) = 0.5 (нейтральный старт)
        self.thought_gates = nn.Parameter(torch.zeros(max_thoughts))

        # Позиционные векторы для каждой мысли
        # Маленькая инициализация 0.01 — не мешать первому проходу
        self.thought_pos = nn.Parameter(torch.randn(max_thoughts, hdc_dim) * 0.01)

        # Нормализация перед дополнительными мыслями
        self.ln = nn.LayerNorm(hdc_dim)

    def forward(self, h, blocks, n_thoughts=3, use_checkpoint=False):
        n_thoughts = min(n_thoughts, self.max_thoughts)

        # === Мысль 1: обязательная ===
        for block in blocks:
            if use_checkpoint and self.training:
                h = torch_checkpoint(block, h, use_reentrant=False)
            else:
                h = block(h)

        if n_thoughts <= 1:
            return h

        # === Мысли 2, 3, ...: дополнительные ===
        for t in range(1, n_thoughts):
            gate = torch.sigmoid(self.thought_gates[t])
            # gate: scalar [0..1]

            # Входной сигнал для этой мысли:
            # нормализуем текущий h + добавляем позицию мысли
            thought_input = self.ln(h) + self.thought_pos[t]

            # Прогоняем через ТЕ ЖЕ 8 блоков
            thought = thought_input
            for block in blocks:
                if use_checkpoint and self.training:
                    thought = torch_checkpoint(block, thought, use_reentrant=False)
                else:
                    thought = block(thought)

            # Гейтированная интерполяция:
            # "насколько слушать эту мысль?"
            h = h + gate * (thought - h)

        return h
```

### Визуализация: как меняется h

Представим вектор `h` в 2D (на самом деле 4096D, но идея та же):

```
         h3 (финал)
          *
         /|
        / |
  gate  / |
       /  |
      *   * h1 (после первой мысли)
     h2 (после второй мысли)
```

- После первой мысли: h1 — некоторая точка в пространстве
- После второй мысли: h2 — другая точка (блоки "передумали")
- gate определяет: движемся от h1 к h2 или остаёмся у h1
- h (финальный) = `(1-gate) × h1 + gate × h2`

Если gate = 0 — h не меняется (вторая мысль ничего не добавила).
Если gate = 1 — h = h2 (полностью "согласились" с новым результатом).

---

### Почему именно 3 мысли

Это не случайный выбор. Это результат экспериментов.

```
1 мысль:  loss стагнирует высоко — модель не обучается нормально
2 мысли:  лучше, но есть "плато" — застреваем
3 мысли:  прорыв! — loss реально падает, качество растёт
4 мысли:  потенциально лучше, но дороже в 1.33×, мало дополнительного эффекта
```

Зафиксировали 3 как оптимальный баланс.

**Гипотеза: почему 2 недостаточно**

Когда система обрабатывает что-то дважды — это одна точка обратной связи.
Три прохода — две точки. Между двумя точками уже можно установить "направление движения",
можно стабилизировать результат.

Аналогия с математикой: для нахождения экстремума функции нужно минимум 3 точки
(чтобы понять "идём вниз или вверх").

### Сколько параметров стоит ThoughtLoop

```
thought_gates: max_thoughts = 4 параметра    (4 числа!)
thought_pos:   4 × 4096     = 16,384 параметров
LayerNorm:     2 × 4096     =  8,192 параметров
──────────────────────────────────────────────
ИТОГО: 24,580 параметров (0.008% всей модели)
```

25 тысяч параметров.

Но ThoughtLoop **переиспользует** 168 миллионов параметров (все 8 блоков) трижды.

Это самый "дешёвый" способ утроить вычислительную мощность:
- Без ThoughtLoop: 8 × 21M = 168M параметров, 1 проход
- С ThoughtLoop:  8 × 21M = 168M параметров, 3 прохода = в 3 раза больше вычислений

Платим вычислительным временем, не параметрами.

### Важно: torch.compile не работает

В коде есть такой комментарий:

```python
# torch.compile disabled: causes infinite recompilation with 3 thoughts
# ThoughtLoop has dynamic control flow that breaks CUDA graphs
print("torch.compile: OFF (incompatible with Thought Loops)")
```

`torch.compile` — это оптимизация PyTorch которая компилирует вычислительный граф
в быстрый CUDA-код. Обычно даёт 1.5-2× ускорение.

Но: `torch.compile` требует **фиксированный граф** — одинаковые операции в одинаковом порядке.

У нас есть Python-цикл: `for t in range(1, n_thoughts)`.
При каждом изменении `n_thoughts` граф меняется — compile перекомпилирует заново.

Это мы обнаружили болезненно: процесс завис на 73 минуты при 100% GPU,
ничего не записывая в лог — бесконечная рекомпиляция.

Если понадобится compile: нужно "развернуть" цикл руками (три отдельные функции).

### Формула ThoughtLoop

```
Мысль 1:  h₁ = Blocks(x)
Мысль t (t > 1):
  input_t = LayerNorm(h_{t-1}) + pos_t
  h_t     = Blocks(input_t)
  g_t     = sigmoid(gate_t)
  h       = h_{t-1} + g_t × (h_t - h_{t-1})
           = (1 - g_t) × h_{t-1} + g_t × h_t
```

Вот что каждый символ означает буквально:
- `Blocks(·)` — 8 блоков HDCBlock последовательно
- `pos_t` — обучаемый вектор "позиции мысли" (16K params)
- `gate_t` — обучаемый scalar [0..1] — "вес этой мысли"
- `(1 - g_t) × h + g_t × h_t` — интерполяция между старым и новым

### Итоги главы

Thought Loops — многопроходное мышление:

```
Одни 8 блоков, применённые 3 раза:

Мысль 1: базовый анализ
Мысль 2: уточнение     (с learned gate)
Мысль 3: финализация   (с learned gate)
```

Ключевые части:
- **Thought Position Embeddings**: каждый проход получает свой сигнал
- **Learned Gates**: модель сама решает вес каждого прохода
- **3 мысли**: 1 мало, 2 — плато, 3 — прорыв (эмпирически)

Стоимость: 24K параметров — переиспользуют 168M параметров блоков трижды.

Ограничение: torch.compile несовместим (динамический цикл).
