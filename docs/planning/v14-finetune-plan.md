## Два подхода к файнтюну

### Вариант A: LoRA на Controller

Стандартный LoRA — замораживаем base модель, добавляем low-rank адаптеры.

**Куда вешать:**
- **Controller (down/up)** — основная цель, самая тяжёлая часть (~167M params из 299M)
- **Codebook** — заморозить полностью
- **Binding vectors** — уже крошечные (12K params/слой), LoRA не нужен
- **Memory (mass/decay)** — маленькие, можно заморозить

**Параметры:**
- LoRA rank: 8-16
- Обучаемых: ~1-2M params (вместо 299M)
- Размер адаптера: ~5-10 MB

**Плюсы:**
- Проверенный подход
- Base модель не портится
- Можно делать разные адаптеры (диалог, код, перевод)

### Вариант B: HDC-native адаптер (ЭКСПЕРИМЕНТАЛЬНЫЙ)

Один bipolar binding vector (4096 бит) на блок. `bind(h, adapter)` = поэлементное умножение на ±1.

**Механика:**
```python
# В каждом блоке, перед Controller:
adapter = self.adapter_vectors[block_idx]  # (4096,) bipolar ±1
h = h * adapter  # bind = переориентация пространства
# ... Controller, Memory, Attention работают как обычно
```

**Параметры:**
- 8 блоков × 4096 бит = **4 KB** адаптер
- 2^4096 возможных конфигураций — пространство огромное
- Обучение: можно через STE или binary voting

**Плюсы:**
- 4 KB вместо 5-10 MB
- HDC-native — философски согласовано с архитектурой
- Сотни адаптеров на телефоне
- bind — изоморфизм, не теряет информацию

**Риски:**
- Одна степень свободы на блок — может не хватить выразительности
- Не проверено ни на одной реальной задаче

## План экспериментов

1. Сначала HDC-native адаптер (4 KB) — быстрый тест на MacBook
2. Если не хватит выразительности → LoRA rank-8 на Controller
3. Если LoRA работает → попробовать гибрид: HDC-adapter + LoRA rank-4
