# Idea: Three-Phase Training with Minimal Internal Language

**Date**: 2026-04-07
**Status**: Concrete training plan, needs language design
**Author**: Oleg (ideation), Claude (structuring)

## Core Idea

Train model in three phases, like a child:
1. Learn to THINK (internal language)
2. Learn to UNDERSTAND (text through logic)
3. Learn to COMMUNICATE (instructions, dialogue)

Use LLM to generate training datasets for each phase.

## The Internal Language: Minimal Operator Set

**Oleg's key insight**: NOT domain-specific operators (CAPITAL_OF, BOILING_POINT).
That makes the language enormous and is just disguised knowledge.

Instead: ~10-15 ABSTRACT operators. Like assembly language — 
few instructions, but can express anything.

### Proposed Minimal Operator Set (~12 operators)

```
// Identity & Classification
IS(X, Y)          — X is Y (equivalence, identity)
TYPE(X, Y)        — X is a kind/type of Y

// Relations
REL(X, Y, R)      — X relates to Y with role R
HAS(X, Y)         — X contains/possesses Y
PART(X, Y)        — X is part of Y

// Logic
AND(A, B)          — both A and B
OR(A, B)           — A or B  
NOT(A)             — negation

// Causality & Sequence  
CAUSE(A, B)        — A leads to / causes B
SEQ(A, B)          — A then B (temporal order)

// Queries & Variables
FIND(X, condition) — search for X where condition is true
VAL(X, V)          — X has value V

// Meta
DEF(name, expr)    — define new operator from existing ones
```

**That's it. ~12 primitives.**

R in REL() is not a fixed label — it's a VECTOR in HDC space.
The model learns what relations exist, the language only provides
the structure REL(X, Y, something).

### How "capital of France" works with minimal operators

```
// Knowledge (learned in Phase 2):
REL(Paris, France, r1)    — Paris relates to France with some relation r1
TYPE(Paris, city)          — Paris is a type of city
TYPE(France, country)      — France is a type of country

// The vector r1 is LEARNED, not named "capital_of"
// It's just a point in HDC space that the model associates 
// with the pattern of "main city of country"

// Query:
FIND(X, AND(TYPE(X, city), REL(X, France, r1)))
→ X = Paris
```

### Self-Expansion

The model can CREATE new operators using DEF:

```
// Model discovers a recurring pattern:
// REL(X, Y, r1) AND TYPE(X, city) AND TYPE(Y, country)
// appears often, so it defines:

DEF(capital, FIND(X, AND(TYPE(X, city), REL(X, $Y, r1))))

// Now "capital" is a macro, not a hardcoded operator
// The model GREW its language, we didn't add it
```

In theory, for consciousness and self-development,
the model should independently expand its own language
by noticing recurring patterns and crystallizing them into new operators.

## Three Training Phases

### Phase 1: Learn Internal Language
**Dataset**: Generated by LLM (Claude/GPT)
**Size**: ~500K-1M examples
**Format**: text → internal language translation

```
"cats are animals"
→ TYPE(cat, animal)

"rain causes floods"
→ CAUSE(rain, flood)

"the book is on the table and the pen is in the box"
→ AND(REL(book, table, r_on), REL(pen, box, r_in))

"who wrote Romeo and Juliet?"
→ FIND(X, REL(X, romeo_and_juliet, r_wrote))
```

Relations (r_on, r_in, r_wrote) are HDC vectors, not string labels.
During training they become points in space.

### Phase 2: Text Understanding Through Logic
**Dataset**: Regular text + "thinking" annotations (generated by LLM)
**Format**: input → [think in internal language] → understanding

```
Input: "Tokyo is the largest city in Japan"
Think: IS(Tokyo, city) AND REL(Tokyo, Japan, r_location) AND VAL(Tokyo, largest)

Input: "Because it rained, the game was cancelled"  
Think: CAUSE(rain, NOT(game))

Input: "If you study hard, you will pass the exam"
Think: CAUSE(study, pass_exam)
```

Model learns to BUILD logical representations from raw text.

### Phase 3: Instruction Tuning with Internal Reasoning
**Dataset**: Q&A pairs with internal reasoning chain
**Format**: question → [internal reasoning] → answer

```
User: "What is the capital of Japan?"
[internal]: FIND(X, AND(TYPE(X, city), REL(X, Japan, r_capital)))
[internal]: SEARCH → X = Tokyo  
[internal]: TEMPLATE(IS(Tokyo, answer))
Output: "The capital of Japan is Tokyo"

User: "Why do leaves fall in autumn?"
[internal]: FIND(reason, CAUSE(reason, AND(REL(leaves, tree, r_part), SEQ(autumn, fall))))
[internal]: SEARCH → reason = less_sunlight → less_chlorophyll → ...
Output: "Leaves fall because..."
```

## At Inference Time

Model OPERATES in its internal language:
```
Input text → Parse to operators → Reason → Generate operators → Decode to text
```

The internal language is the "mentalese" — the thinking medium.
Text is just input/output interface.

## What LLM Generates (dataset creation)

Ask Claude/GPT:
"Given this sentence, express its logical structure using ONLY these operators:
IS, TYPE, REL, HAS, PART, AND, OR, NOT, CAUSE, SEQ, FIND, VAL, DEF"

LLM understands logic well and can generate millions of examples.
We don't need to hand-craft the dataset.

## Advantages

1. **Language is tiny** — 12 operators fit in small model
2. **Knowledge is separate** — stored as vectors (r1, r2...), not in operators
3. **Self-expanding** — DEF allows model to grow its vocabulary
4. **Verifiable** — internal reasoning can be checked formally
5. **Language-independent** — same operators work for English, Russian, any language
6. **LLM bootstrapped** — no manual dataset creation needed

## Risks

1. Model may learn to IMITATE format without real understanding
   - Mitigation: test on novel reasoning tasks not in training data
2. 12 operators may not be enough for complex reasoning
   - Mitigation: DEF allows self-expansion
3. Translating between text and operators may be lossy
   - Mitigation: keep vector relations (R) rich and high-dimensional
4. Training three phases is complex — each phase depends on previous
   - Mitigation: start small, validate each phase before moving on
